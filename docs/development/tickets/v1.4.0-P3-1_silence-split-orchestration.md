## 1. タスク目的とゴール

`InferenceAudioGenerator.GenerateAudioWithSilenceSplitAsync`（L274-342）は、沈黙句分割・反復推論・結果結合というオーケストレーションロジックを推論クラス内に持っている。これは単一責任原則に反しており、`InferenceAudioGenerator`が推論以外の責務を抱える原因となっている。

本チケットでは、このオーケストレーションロジックを新規クラス `SplitInferenceOrchestrator` に移動し、`PiperTTS.Inference.cs`の呼び出し元をオーケストレーター経由に変更する。

**ゴール**:
- 沈黙句分割のオーケストレーションロジックが `SplitInferenceOrchestrator` に分離されている
- `PiperTTS.Inference.cs` がオーケストレーター経由でSilence分割を実行している
- `InferenceAudioGenerator` の `GenerateAudioWithSilenceSplitAsync` はまだ残存（P3-2で削除）
- 既存の動作に変更がなく、全テストが通過する

## 2. 実装する内容の詳細

### 2.1 SplitInferenceOrchestratorクラスの作成

**ファイル**: `Assets/uPiper/Runtime/Core/AudioGeneration/SplitInferenceOrchestrator.cs`（新規作成）

候補B（独立オーケストレータークラス）を採用する。`IInferenceAudioGenerator` を受け取り、各フレーズに対して `GenerateAudioAsync` または `GenerateAudioWithProsodyAsync` を呼ぶ。

```csharp
using System;
using System.Collections.Generic;
using System.Threading;
using System.Threading.Tasks;
using uPiper.Core.Logging;

namespace uPiper.Core.AudioGeneration
{
    /// <summary>
    /// 沈黙句分割のオーケストレーションを担当する。
    /// 音素列を沈黙トークンの位置で分割し、句ごとに独立推論を行い、
    /// 句間にゼロサンプルの無音区間を挿入して結合する。
    /// </summary>
    public class SplitInferenceOrchestrator
    {
        private readonly IInferenceAudioGenerator _generator;

        public SplitInferenceOrchestrator(IInferenceAudioGenerator generator)
        {
            _generator = generator ?? throw new ArgumentNullException(nameof(generator));
        }

        /// <summary>
        /// 沈黙句分割付きで音声を生成する。
        /// </summary>
        public async Task<float[]> GenerateWithSilenceSplitAsync(
            int[] phonemeIds,
            int[] prosodyA1,
            int[] prosodyA2,
            int[] prosodyA3,
            Dictionary<string, float> phonemeSilence,
            Dictionary<string, int> phonemeIdMap,
            int sampleRate,
            float lengthScale = 1.0f,
            float noiseScale = 0.667f,
            float noiseW = 0.8f,
            int speakerId = 0,
            int languageId = 0,
            CancellationToken cancellationToken = default)
        {
            // ... 実装詳細は2.1(a)-(d)参照
        }
    }
}
```

> `sampleRate` はパラメータとして明示的に渡す設計。`IInferenceAudioGenerator.SampleRate` 経由でも取得可能だが、オーケストレーターが推論器の内部状態に依存しない方が責務が明確になる。

**(a) フレーズ分割（L291-293相当の移植）**:

```csharp
var phrases = PhonemeSilenceProcessor.SplitAtPhonemeSilence(
    phonemeIds, prosodyA1, prosodyA2, prosodyA3,
    phonemeSilence, phonemeIdMap, sampleRate);

PiperLogger.LogInfo(
    $"[SplitInferenceOrchestrator] Silence split: {phrases.Count} phrases from {phonemeIds.Length} phonemes");
```

- `PhonemeSilenceProcessor.SplitAtPhonemeSilence` は静的メソッドのためそのまま呼び出し可能
- `sampleRate` をパラメータから受け取る

**(b) フレーズごとの推論（L301-323相当の移植）**:

```csharp
var segments = new List<(float[] Audio, int SilenceSamples)>();
var totalLength = 0;

for (var p = 0; p < phrases.Count; p++)
{
    var phrase = phrases[p];
    if (phrase.PhonemeIds == null || phrase.PhonemeIds.Length == 0)
        continue;

    cancellationToken.ThrowIfCancellationRequested();

    float[] phraseAudio;
    var hasProsody = phrase.ProsodyA1 != null && _generator.SupportsProsody;

    if (hasProsody)
    {
        phraseAudio = await _generator.GenerateAudioWithProsodyAsync(
            phrase.PhonemeIds,
            phrase.ProsodyA1, phrase.ProsodyA2, phrase.ProsodyA3,
            lengthScale, noiseScale, noiseW,
            speakerId, languageId,
            cancellationToken);
    }
    else
    {
        phraseAudio = await _generator.GenerateAudioAsync(
            phrase.PhonemeIds,
            lengthScale, noiseScale, noiseW,
            speakerId, languageId,
            cancellationToken);
    }

    segments.Add((phraseAudio, phrase.SilenceSamples));
    totalLength += phraseAudio.Length + phrase.SilenceSamples;

    PiperLogger.LogDebug(
        $"[SplitInferenceOrchestrator] Phrase {p + 1}/{phrases.Count}: " +
        $"{phraseAudio.Length} samples + {phrase.SilenceSamples} silence");
}
```

**ポイント**:
- `IInferenceAudioGenerator` の公開メソッド（`GenerateAudioAsync` / `GenerateAudioWithProsodyAsync`）を呼ぶ
- 各メソッドが内部で `UnityMainThreadDispatcher.RunOnMainThreadAsync` を使用するため、オーケストレーターはThread非依存
- `lock(_lockObject)` は `InferenceAudioGenerator` 内部にあるため、オーケストレーターから直接参照不要

**(c) セグメント結合（L331-338相当の移植）**:

```csharp
var result = new float[totalLength];
var offset = 0;
foreach (var (audio, silenceSamples) in segments)
{
    Array.Copy(audio, 0, result, offset, audio.Length);
    offset += audio.Length;
    offset += silenceSamples; // Zero-initialized (silence)
}

PiperLogger.LogInfo(
    $"[SplitInferenceOrchestrator] Silence split complete: " +
    $"{totalLength} total samples ({segments.Count} phrases)");
return result;
```

**(d) 空フレーズ時の早期リターン**:

```csharp
if (phrases.Count == 0)
{
    PiperLogger.LogWarning("[SplitInferenceOrchestrator] No phrases after silence split");
    return Array.Empty<float>();
}
```

## 5. 懸念事項・レビュー項目

### 5.1 スレッドセーフティ

`SplitInferenceOrchestrator` 自体はステートレスであり、`_generator` への参照のみを保持する。`IInferenceAudioGenerator` の各メソッドが内部で `UnityMainThreadDispatcher.RunOnMainThreadAsync` と `lock(_lockObject)` を使用しているため、オーケストレーターがスレッドセーフティを直接管理する必要はない。

ただし、複数の `GenerateWithSilenceSplitAsync` 呼び出しが同時に行われた場合、フレーズごとの推論が `lock(_lockObject)` でシリアライズされる。これは既存動作と同一であり、問題ない。

### 5.2 旧メソッドとの共存期間

本チケット完了後、以下の状態になる:
- `PiperTTS.Inference.cs` → `SplitInferenceOrchestrator` 経由（新パス）
- `IInferenceAudioGenerator.GenerateAudioWithSilenceSplitAsync` → インターフェースに残存
- `InferenceAudioGenerator.GenerateAudioWithSilenceSplitAsync` → 実装に残存（未使用）

P3-2で旧メソッドに `[Obsolete]` を付与し、インターフェースと実装から削除する。この共存期間中に外部利用者が旧メソッドを直接呼んでいる場合はP3-2で移行が必要。

### 5.3 sampleRate の渡し方

オーケストレーターの `GenerateWithSilenceSplitAsync` メソッドに `sampleRate` をパラメータとして渡す設計を採用した。代替案として `IInferenceAudioGenerator.SampleRate` プロパティ経由で取得する方法もあるが、以下の理由でパラメータ方式を選択:

- オーケストレーターが推論器の内部状態に依存しない
- テスト時にsampleRateを明示的に指定できる
- 将来的にsampleRateが呼び出し元で決定される可能性がある

### 5.4 レビュー重点項目

- [ ] `SplitInferenceOrchestrator` がProsody有無で正しいメソッド（`GenerateAudioAsync` / `GenerateAudioWithProsodyAsync`）を呼び分けているか
- [ ] フレーズ結合時の無音区間（`silenceSamples`）がゼロ初期化されたfloat配列として正しく挿入されているか
- [ ] `PiperTTS.Inference.cs` の2箇所の呼び出し元が正しくオーケストレーター経由に変更されているか
- [ ] `DisposeInferenceResources` で `_splitOrchestrator` が `_inferenceGenerator.Dispose()` より前にnull化されているか
- [ ] `CancellationToken` がフレーズループ内で正しくチェックされているか
