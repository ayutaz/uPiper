## 1. タスク目的とゴール

`IInferenceAudioGenerator` は現在3つの音声生成メソッド（`GenerateAudioAsync`, `GenerateAudioWithProsodyAsync`, `GenerateAudioWithSilenceSplitAsync`）を持っている。P3-1でSilence分割がオーケストレーション層に移動済みのため、インターフェースを整理して2メソッドに統合する。

**統合方針**:
- `GenerateAudioAsync` と `GenerateAudioWithProsodyAsync` を1メソッドに統合（prosodyA1/A2/A3をnullable化）
- `GenerateAudioWithSilenceSplitAsync` をインターフェースから除去（P3-1でオーケストレーション層に移動済み）

**ゴール**:
- `IInferenceAudioGenerator` が `InitializeAsync`（2オーバーロード）+ `GenerateAudioAsync`（統合版）= 3メソッドのシンプルなインターフェースになる
- Prosody有無の分岐が呼び出し元（`PiperTTS.Inference.cs`）から消え、統合メソッドに一本化される
- 破壊的変更が段階的に管理され、移行期間中は `[Obsolete]` 警告で通知される
- 全テストが通過する

### 2.3 ステップ2（実装側）: InferenceAudioGeneratorの統合メソッド

**ファイル**: `Assets/uPiper/Runtime/Core/AudioGeneration/InferenceAudioGenerator.cs`

統合メソッドの実装:

```csharp
/// <inheritdoc/>
public async Task<float[]> GenerateAudioAsync(
    int[] phonemeIds,
    int[] prosodyA1 = null,
    int[] prosodyA2 = null,
    int[] prosodyA3 = null,
    float lengthScale = 1.0f,
    float noiseScale = 0.667f,
    float noiseW = 0.8f,
    int speakerId = 0,
    int languageId = 0,
    CancellationToken cancellationToken = default)
{
    ValidateGenerationPrerequisites(phonemeIds);

    var hasProsody = prosodyA1 != null || prosodyA2 != null || prosodyA3 != null;
    if (hasProsody && !_supportsProsody)
    {
        PiperLogger.LogWarning(
            "[InferenceAudioGenerator] Prosody data provided but model does not support prosody. Ignoring prosody.");
        prosodyA1 = null;
        prosodyA2 = null;
        prosodyA3 = null;
    }

    return await UnityMainThreadDispatcher.RunOnMainThreadAsync(() =>
    {
        lock (_lockObject)
        {
            return ExecuteInference(
                phonemeIds, prosodyA1, prosodyA2, prosodyA3,
                lengthScale, noiseScale, noiseW,
                speakerId, languageId);
        }
    });
}
```

**ポイント**:
- Prosody有無の判定を統合メソッド内で行う
- モデルがProsody非対応でProsodyデータが渡された場合は警告ログを出力しnull化（既存の `GenerateAudioWithProsodyAsync` がthrowしていたのとは異なり、graceful degradation）
- `ExecuteInference` は既にprosodyA1=nullの場合を正しく処理する（L431 `hasAnyProsodyInput` チェック）

### 5.2 シグネチャ変更による位置引数の問題

旧 `GenerateAudioAsync(int[] phonemeIds, float lengthScale, ...)` を統合シグネチャ `GenerateAudioAsync(int[] phonemeIds, int[] prosodyA1, ...)` に置き換えると、位置引数で `lengthScale`（float）が `prosodyA1`（int[]）に渡されるケースでコンパイルエラーになる。

**影響**: float→int[]の暗黙変換は存在しないため、コンパイルエラーとして検出される。実行時の不具合にはならない。

**対応**: 呼び出し元で名前付き引数を使用するか、prosody引数を明示的にnullで渡す。

```csharp
// 方法1: 名前付き引数
await generator.GenerateAudioAsync(phonemeIds, lengthScale: 1.0f, noiseScale: 0.667f);

// 方法2: null明示
await generator.GenerateAudioAsync(phonemeIds, null, null, null, 1.0f, 0.667f);
```

### 5.3 graceful degradation vs 例外

既存の `GenerateAudioWithProsodyAsync` はProsody非対応モデルで `InvalidOperationException` をスローする。統合メソッドではgraceful degradation（警告ログ+Prosody無視）を採用する。これは以下の理由による:

- 統合メソッドはProsody有無を呼び出し元が意識しない設計
- Prosody非対応モデルでProsodyデータが渡されるのは正常なフロー（多言語Phonemizerが常にProsodyを返す場合等）
- 例外スローは呼び出し元にProsody有無の事前チェックを強制し、統合の目的に反する

### 5.4 テスト修正の規模

`InferenceAudioGeneratorMultilingualTests.cs` に60+テストがあり、その多くが `GenerateAudioWithProsodyAsync` を直接呼んでいる。機械的な置換（メソッド名変更）で対応可能だが、テストの意図（Prosodyパスの検証）が失われないよう注意する。

**推奨**: テスト名に`WithProsody`が含まれるケースは、統合メソッドにprosody引数を明示的に渡す形に変更し、テストの意図を保持する。
