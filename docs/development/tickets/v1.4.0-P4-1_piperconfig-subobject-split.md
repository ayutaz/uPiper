## 1. タスク目的とゴール

`PiperConfig.cs` は現在21フィールドが単一クラスにフラットに並んでおり、設定の論理グループが不明瞭になっている。これを4つのサブオブジェクトに分割し、設定の関心ごとの凝集度を高める。

**設計方針**: 選択肢Bを採用する。v1.4.0ではサブオブジェクトを`private`（内部分割のみ）とし、公開APIは現行のフラットプロパティを維持する。`PiperConfig`自体がファサードとして残り、後方互換プロパティ（`public string DefaultLanguage => General.DefaultLanguage;` 等）で委譲する。これにより破壊的変更を最小化する。

**ゴール**:
- 4つのサブオブジェクトクラス（`GeneralSettings`, `InferenceSettings`, `AudioSettings`, `CacheSettings`）が作成されている
- `PiperConfig` がファサードとして全フィールドをサブオブジェクトに委譲
- `Validate()` が各サブオブジェクトの `Validate()` に委譲
- 公開APIに変更なし（後方互換維持）
- 全テストパス

### 2.2 PiperConfig のファサード化

**ファイル**: `Assets/uPiper/Runtime/Core/PiperConfig.cs`

PiperConfigの既存フィールドをサブオブジェクトに移動し、後方互換プロパティで委譲する。

```csharp
[Serializable]
public class PiperConfig
{
    // サブオブジェクト（privateでシリアライズ対象）
    [SerializeField] private GeneralSettings _general = new();
    [SerializeField] private InferenceSettings _inference = new();
    [SerializeField] private AudioSettings _audio = new();
    [SerializeField] private CacheSettings _cache = new();

    // === 後方互換プロパティ（General） ===
    public bool EnableDebugLogging
    {
        get => _general.EnableDebugLogging;
        set => _general.EnableDebugLogging = value;
    }
    public string DefaultLanguage
    {
        get => _general.DefaultLanguage;
        set => _general.DefaultLanguage = value;
    }
    // ... 他のフィールドも同様

    // === Validate() はサブオブジェクトに委譲 ===
    public void Validate()
    {
        _cache.Validate();
        _audio.Validate();
        _inference.Validate();
        _general.Validate();
        PiperLogger.LogInfo("PiperConfig validated successfully");
    }
}
```

**重要**: `[Header("General Settings")]` 等のInspector用属性はサブオブジェクト側に移動する。ただし選択肢B（privateサブオブジェクト）ではInspector表示が変わるため、カスタムエディタまたは後方互換プロパティに `[Header]` を付与して既存のInspector表示を維持する。

### 2.3 Validate() 委譲順序

バリデーション順序は現行と同一を維持する:
1. `_cache.Validate()` — キャッシュサイズ検証（L214-224）
2. `_audio.Validate()` — サンプルレート検証（L226-235）、RMS検証（L288-301）、PhonemeSilence形式検証（L310-321）
3. `_inference.Validate()` — WorkerThreads検証（L237-251）、タイムアウト検証（L265-274）、バッチサイズ検証（L276-286）、ウォームアップ検証（L303-308）、GPU検証（L327-328）
4. `_general.Validate()` — 言語コード検証（L253-263）

選択肢B（後方互換プロパティ維持）のため、以下の参照箇所は **変更不要**:

## 5. 懸念事項・レビュー項目

### 5.1 シリアライズ互換性

選択肢B（privateサブオブジェクト + `[SerializeField]`）では、既存のシリアライズ済みPiperConfigアセットが新しいサブオブジェクト構造を認識しない。

**影響**: ユーザーがScriptableObject等でPiperConfigをシリアライズしている場合、v1.4.0アップデート時にデフォルト値にリセットされる可能性がある。ただし、現状PiperConfigは主にコード上で `new PiperConfig()` や `PiperConfig.CreateDefault()` で生成されており、シリアライズ済みアセットとしての使用は限定的と想定。

**緩和策**: CHANGELOG/リリースノートで「PiperConfigのシリアライズ形式が変更されました。Inspector上の設定値を再確認してください」と明記する。

### 5.2 WorkerThreads=0 自動検出の副作用

`InferenceSettings.Validate()` 内の `WorkerThreads == 0` 判定（L243-246）では `SystemInfo.processorCount` を参照してフィールドを書き換える副作用がある。これは本チケットのスコープ外だが、サブオブジェクト化により副作用の所在が明確になる（`InferenceSettings.Validate()` に局所化される）。

### 5.3 CreateDefault() メソッド

`PiperConfig.CreateDefault()` 内の IL2CPP 最適化（L200-203）は `_inference.WorkerThreads` と `_cache.MaxCacheSizeMB` を設定する。後方互換プロパティのsetterを経由するため、変更不要。

### 5.4 レビュー重点項目

- [ ] 後方互換プロパティのget/setが全21フィールドに対して漏れなく定義されているか
- [ ] Validate()の委譲順序が現行と同一か（テスト結果への影響を防止）
- [ ] 定数が正しいサブオブジェクトに配置されているか
- [ ] `[Header]` 属性の配置がInspector表示を破壊していないか
- [ ] `[NonSerialized]` 属性の移動が正しいか（P1-3完了前提: ParsedPhonemeSilence削除済み）

## 実装メモ（2026-04-06完了）

PiperConfig.cs自体のフィールド分割（Inspector互換維持のため）は行わず、
**ValidatedPiperConfig** 不変スナップショットとして実装。

- 新規: `Assets/uPiper/Runtime/Core/ValidatedPiperConfig.cs`
- `PiperConfig.ToValidated()` メソッド追加（Validate()を内包）
- `ValidatedPiperConfig` が `ParsedPhonemeSilence: IReadOnlyDictionary<string, float>` を保持
- `PiperTTS._parsedPhonemeSilence` フィールド削除（ValidatedPiperConfigに統合）
- `PiperTTS._validatedConfig` フィールド追加

この方式により PiperConfig の [Serializable] + Unity Inspector 互換性を維持しつつ、
バリデーション後の値を不変オブジェクトとして扱える設計を実現した。
