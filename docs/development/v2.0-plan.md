# uPiper v2.0 計画

**作成日**: 2026-04-06
**ベース**: architecture-improvement-roadmap.md v2.0セクション + v1.4.0設計レビュー追加知見
**前提**: v1.4.0 が develop にマージ完了

---

## 概要

v2.0 はセマンティックバージョニング上の **メジャー破壊的変更リリース** である。
v1.3.x/v1.4.0 で段階的に準備してきた以下の方向性を完遂する:

1. **static 依存の排除** -- PuaTokenMapper インスタンス化、テスト間状態リーク根絶
2. **MultilingualPhonemizer の Strategy パターン完全移行** -- `ILanguageG2PHandler` による言語プラグイン化
3. **piper-plus データモデル互換** -- `int[]` PhonemeIdMap、Prosody フラット配列、pua.json ランタイム読み込み
4. **設定・API の整理** -- ValidatedPiperConfig ネスト化、IPiperConfigReadOnly、Obsolete 削除

全16項目を4フェーズに分類し、フェーズ内は並行実施可能、フェーズ間は依存順に実施する。

---

## Phase 1: 基盤リファクタリング（破壊的変更の土台）

### P1-1: PuaTokenMapper インスタンスクラス化

**目的**: テスト間状態リークの根本解決 + DI 対応

**現状**: `PuaTokenMapper` は `static class` であり、`ConcurrentDictionary<string, char> Token2Char` / `ConcurrentDictionary<char, string> Char2Token` を static フィールドとして保持。`ResetForTesting()` / `ResetOnDomainReload()` で手動リセットが必要。

**変更内容**:
- `static class` を通常の `class` に変更
- 固定マッピング（96エントリ）は `static readonly IReadOnlyDictionary<string, int> FixedPuaMapping` のまま保持
- 動的割り当て部分（`_nextDynamic`, `Token2Char`, `Char2Token`）をインスタンスフィールドに移動
- コンストラクタで固定マッピングを初期化、`ResetForTesting()` 不要に
- `MultilingualPhonemizer`, `PhonemeEncoder` はコンストラクタ注入で受け取る

**影響範囲**: `MultilingualPhonemizer`, `PhonemeEncoder`, `PiperTTS.Inference.cs`, 全テストクラス

**breaking change**: public API シグネチャ変更（static メソッド → インスタンスメソッド）

---

### P1-2: pua.json ランタイム読み込み

**目的**: piper-plus との仕様同期の自動化

**現状**: 固定マッピングは C# ハードコード。piper-plus 側の `pua.json` との整合性は CI テスト（pua.json クロスバリデーション）で検出。

**変更内容**:
- `PuaTokenMapper` に `InitializeAsync(string json)` / `Initialize(string json)` を追加
- `StreamingAssets/uPiper/pua.json` から読み込み（ファイルなしの場合はハードコード固定マッピングにフォールバック）
- WebGL: `UnityWebRequest` 経由の非同期読み込み必須

**依存**: P1-1（インスタンスクラス化後に実施）

**WebGL 制約**: StreamingAssets は非同期読み込み必須。`PuaTokenMapper.InitializeAsync()` を `MultilingualPhonemizer.InitializeAsync()` 内で呼び出す。

---

### P1-3: MultilingualPhonemizer Dictionary Registry 化

**目的**: `_ownsXxx` フラグ7つ（`_ownsJa`, `_ownsEn`, `_ownsEs`, `_ownsFr`, `_ownsPt`, `_ownsZh`, `_ownsKo`）と個別フィールド7つの廃止

**現状**: 7言語分のエンジンフィールド + 所有権フラグが個別に存在。言語追加時にフィールド2つ + InitializeAsync 分岐 + Dispose 分岐を追加する必要がある。

**変更内容**:
- `Dictionary<string, ILanguageG2PHandler>` をレジストリとして導入
- 各エンジンを `ILanguageG2PHandler` 実装で wrap
- 所有権は `ILanguageG2PHandler.IsOwned` プロパティ、または登録時のファクトリパターンで管理
- `InitializeAsync` は未登録言語に対してデフォルトファクトリから生成
- `Dispose` はレジストリ内の `IsOwned == true` なハンドラのみ Dispose

**依存**: P1-4（ILanguageG2PHandler 定義が先）

---

### P1-4: ILanguageG2PHandler 全面移行

**目的**: 言語追加のスケーラビリティ確保（10言語超対応）

**現状**: v1.4.0 で `switch` + `ProcessXxx()` x 7メソッドに抽出済み。ただし `ProcessXxx` は `MultilingualPhonemizer` のプライベートメソッドのまま。

**変更内容**:
- `ILanguageG2PHandler` インターフェース定義:
  ```
  string LanguageCode { get; }
  bool IsInitialized { get; }
  Task InitializeAsync(CancellationToken ct);
  (string[] Phonemes, int[] A1, int[] A2, int[] A3) Process(string text);
  void Dispose();
  ```
- `ProcessJapanese` → `JapaneseG2PHandler`, `ProcessEnglish` → `EnglishG2PHandler`, ... の7クラスに分離
- `MultilingualPhonemizer.PhonemizeWithProsodyAsync` の switch 文を `_handlers[lang].Process(text)` 呼び出しに置換

**影響範囲**: `MultilingualPhonemizer`, 新規7ハンドラクラス, テスト

---

### P1-5: G2P 全同期化（ProcessFallbackAsync / IPhonemizerBackend 廃止）

**目的**: 非同期パスの排除による簡素化

**現状**: `ProcessFallbackAsync` が `IPhonemizerBackend.PhonemizeAsync` を呼び出す唯一の非同期パス。`IPhonemizerBackend` はレガシーテストスタブ注入のみに使用（`[Obsolete]` 済み）。

**変更内容**:
- `IPhonemizerBackend` インターフェース削除
- `ProcessFallbackAsync` 削除（未対応言語はハンドラ未登録として警告ログのみ）
- `MultilingualPhonemizerOptions.EnPhonemizer` / `KoPhonemizer`（`[Obsolete]` 済み）を削除
- `PhonemizeWithProsodyAsync` → `PhonemizeWithProsody`（同期メソッド化も検討。ただし WebGL の `DotNetG2PPhonemizer.InitializeAsync` があるため、メソッドシグネチャは async のまま維持）

**依存**: P1-4（ハンドラ移行後に旧インターフェースを削除）

---

### P1-6: Obsolete コンストラクタ削除

**目的**: v1.4.0 で `[Obsolete]` 指定済みの旧 API を削除

**現状**: `MultilingualPhonemizer` に `[Obsolete]` コンストラクタ（14引数）が残存。

**変更内容**:
- `MultilingualPhonemizer(IReadOnlyList<string>, string, ...)` コンストラクタを削除
- `MultilingualPhonemizerOptions` 経由のコンストラクタのみ残す

**依存**: P1-4（ハンドラ移行で Options の構造も変わるため同時実施）

## Phase 2: データモデル互換性（piper-plus 準拠）

### P2-1: PhonemeIdMap `int` → `int[]` 型変更

**目的**: piper-plus との完全互換

**現状**: `Dictionary<string, int>` 型。piper-plus は `Dictionary<string, int[]>`（1音素に複数IDを割り当て可能）。現行モデルは全て1要素配列。

**変更内容**:
- `PiperVoiceConfig.PhonemeIdMap` を `Dictionary<string, int[]>` に変更
- `PhonemeEncoder.Encode` / `EncodeWithProsody` の ID 取得ロジックを配列展開に対応
- JSON デシリアライズを `int[]` 対応に変更

**影響範囲**: `PiperVoiceConfig`, `PhonemeEncoder`, `PhonemeSilenceProcessor`, `SplitInferenceOrchestrator`, テスト（15+ファイル）

**breaking change**: `PiperVoiceConfig.PhonemeIdMap` の型変更

---

### P2-2: Prosody フラット配列化

**目的**: piper-plus との推論インターフェース統一

**現状**: A1/A2/A3 を別々の `int[]` として保持・伝播。piper-plus は `int[]` フラット配列（`[a1_0, a2_0, a3_0, a1_1, a2_1, a3_1, ...]`）。

**変更内容**:
- `MultilingualPhonemizeResult` の `ProsodyA1` / `ProsodyA2` / `ProsodyA3` を `int[] ProsodyFlat`（stride=3）に統合
- `SynthesisRequest` のフィールドも同様に統合
- `InferenceAudioGenerator.GenerateAudioAsync` のシグネチャ変更
- テンソル構築ロジック（`PrepareInputs`）を stride=3 展開に変更

**影響範囲**: `MultilingualPhonemizeResult`, `SynthesisRequest`, `IInferenceAudioGenerator`, `InferenceAudioGenerator`, `PhonemeEncoder`, `TTSSynthesisOrchestrator`, `SplitInferenceOrchestrator`, テスト

**breaking change**: `MultilingualPhonemizeResult` / `IInferenceAudioGenerator` の public API 変更

---

### P2-3: NativeArray 統一

**目的**: `ExtractAudioData` → `NormalizeAudio` → `AudioClip.SetData` の中間コピー完全排除

**現状**: `float[]` 中間バッファを経由。`AudioClip.SetData(NativeArray<float>, int)` は Unity 2023.1+ で利用可能。

**変更内容**:
- `InferenceAudioGenerator` の出力を `NativeArray<float>` に変更
- `AudioClipBuilder.BuildAudioClip` を `NativeArray<float>` 受け付けに拡張
- `NormalizeAudioInPlace` を `NativeArray<float>` 対応に拡張

**制約**: Unity 2023.1+ が最低要件。`#if` 分岐を入れる場合は保守コストとの兼ね合い。

**依存**: なし（他項目と独立して実施可能だが、P2-2 の AudioData 型変更と同時が効率的）

## Phase 3: 設定・API 整理

### P3-1: ValidatedPiperConfig ネスト構造化

**目的**: 21プロパティのフラット構造を意味のあるカテゴリに分類

**現状**: `ValidatedPiperConfig` は21個のトップレベルプロパティを持つ。関連性のあるプロパティ群（Language, Performance, Inference, Audio, Silence, General）が平坦に並んでいる。

**変更内容**:
- カテゴリ別の `readonly record struct` を導入:
  - `LanguageSettings` (DefaultLanguage, AutoDetectLanguage, SupportedLanguages, MixedLanguageMode)
  - `PerformanceSettings` (MaxCacheSizeMB, EnablePhonemeCache, WorkerThreads, EnableMultiThreadedInference, InferenceBatchSize)
  - `InferenceSettings` (Backend, EnableWarmup, WarmupIterations, AllowFallbackToCPU, GPUSettings)
  - `AudioSettings` (SampleRate, NormalizeAudio, TargetRMSLevel)
  - `SilenceSettings` (EnablePhonemeSilence, PhonemeSilenceSpec, ParsedPhonemeSilence)
  - `GeneralSettings` (EnableDebugLogging, TimeoutMs)
- `ValidatedPiperConfig` のプロパティをネスト record 経由でアクセス:
  `config.Language.DefaultLanguage`, `config.Inference.Backend` 等

**影響範囲**: `ValidatedPiperConfig`, `TTSSynthesisOrchestrator`, `InferenceAudioGenerator`, `PiperTTS`, テスト

**breaking change**: `ValidatedPiperConfig` の全プロパティアクセスパス変更

---

### P3-2: IPiperConfigReadOnly インターフェース

**目的**: 設定の読み取り専用インターフェースによる不変性保証

**現状**: `PiperConfig` は `[Serializable]` クラスで public フィールドを持つ。`ValidatedPiperConfig` は sealed class だが、インターフェース抽象なし。

**変更内容**:
- `IPiperConfigReadOnly` インターフェースを定義（P3-1 のネスト record をプロパティとして公開）
- `ValidatedPiperConfig` が `IPiperConfigReadOnly` を実装
- 内部コンポーネント（`TTSSynthesisOrchestrator`, `InferenceAudioGenerator` 等）の依存を `IPiperConfigReadOnly` に変更

**依存**: P3-1（ネスト構造化後にインターフェースを定義）

---

### P3-3: Validate() 純粋関数化

**目的**: `PiperConfig.Validate()` の副作用除去

**現状**: `Validate()` は値をクランプ・修正する副作用がある（`WorkerThreads=0` → 自動検出値、`DefaultLanguage` → 小文字正規化等）。`ToValidated()` のXML docに副作用警告を追加済み。

**変更内容**:
- `Validate()` を `ValidatedPiperConfig Validate()` に変更（戻り値で結果を返す、元オブジェクト不変）
- または `static ValidatedPiperConfig CreateValidated(PiperConfig source)` ファクトリメソッドに変更
- `PiperConfig` のフィールドは一切変更しない
- クランプ・正規化は `ValidatedPiperConfig` コンストラクタ内で実施

**依存**: P3-1（ネスト構造と同時に内部ロジック整理）

---

### P3-4: BackendSelector 切り出し

**目的**: `InferenceAudioGenerator.DetermineBackendType` (約90行) の単一責任化

**現状**: プラットフォーム判定 + Metal 回避 + GPUCompute 互換性チェック + WebGL/WebGPU 分岐 + Auto 選択ロジックが1メソッドに凝縮。

**変更内容**:
- `BackendSelector` static クラスを新設（`Runtime/Core/AudioGeneration/BackendSelector.cs`）
- `BackendType Determine(InferenceBackend requested, PlatformInfo platform)` メソッド
- `PlatformInfo` record でプラットフォーム情報をカプセル化（テスト時にモック可能）
- `InferenceAudioGenerator` は `BackendSelector.Determine()` を呼ぶだけに

**影響範囲**: `InferenceAudioGenerator`, 新規 `BackendSelector`, テスト

---

### P3-5: AudioNormalizer 切り出し

**目的**: `AudioClipBuilder` から正規化ロジックを分離

**現状**: `AudioClipBuilder` が `NormalizeAudio` / `NormalizeAudioInPlace` を持つ。AudioClip 構築と正規化は異なる責務。

**変更内容**:
- `AudioNormalizer` static クラスを新設（`Runtime/Core/AudioGeneration/AudioNormalizer.cs`）
- `NormalizeInPlace(float[] audioData, float targetPeak)` / `NormalizeInPlace(NativeArray<float>, float)` メソッド
- `AudioClipBuilder` から正規化メソッドを削除（`BuildAudioClip` のみ残す）
- `TTSSynthesisOrchestrator` が `AudioNormalizer.NormalizeInPlace()` を直接呼び出し

**依存**: P2-3（NativeArray 統一と同時に NativeArray 版を実装すると効率的）

---

### P3-6: SynthesisRequest public API 昇格検討

**目的**: 外部からの合成パラメータ指定の柔軟性向上

**現状**: `SynthesisRequest` は `internal readonly struct`。外部ユーザーは `PiperTTS.GenerateAudioWithInferenceAsync` の個別パラメータでしか指定できない。

**変更内容**:
- `SynthesisRequest` を `public` に変更
- `PiperTTS` に `SynthesizeAsync(SynthesisRequest request)` オーバーロードを追加
- 既存の `GenerateAudioWithInferenceAsync` は内部で `SynthesisRequest` を構築するラッパーとして維持

**判定**: v2.0 で実施するか、ユーザー要望を見て判断。現時点では **検討中** とする。

## Phase 4: 言語検出の高度化

### P4-1: N-gram 言語検出

**目的**: ラテン文字多言語の自動検出

**現状**: `UnicodeLanguageDetector` は Unicode 文字範囲ベース。CJK/ハングル/かな・カタカナは高精度だが、en/es/fr/pt のラテン文字間は `defaultLatinLanguage` に一律フォールバック。

**変更内容**:
- `TrigramLanguageDetector` を新設
- CJK は既存 Unicode 範囲検出（高速・確実）を維持
- ラテン文字間は trigram 頻度分布検出（精度重視）のハイブリッド方式
- 学習データは各言語のコーパスから trigram 頻度テーブルを事前生成し、`StreamingAssets` に配置
- `UnicodeLanguageDetector` のインターフェースを抽出し、`ILanguageDetector` として DI 可能に

**依存**: P1-3 / P1-4（ハンドラ登録後に検出器を差し替え可能にする）

**注意**: 短文（3-5単語）での検出精度が課題。閾値以下のスコアでは `defaultLatinLanguage` にフォールバック。

## 不採用項目

| 提案 | 理由 |
|------|------|
| public API 戻り値 `IMemoryOwner<float>` 化 | `AudioClip.SetData` が `float[]` を要求するため変換コストが発生。費用対効果が見合わない。NativeArray 統一（P2-3）で中間コピーは解消可能 |
| IWarmupStrategy インターフェース | YAGNI。現実的なバリエーションがダミー入力パラメータ違いのみ（v1.3.x で不採用決定済み） |
| AudioBufferPool (TTS 専用プール) | `ArrayPool.Shared` で十分。TTS 呼び出し頻度はリアルタイムレンダリングほどではない（v1.3.x で不採用決定済み） |
| Span/Memory ベース public API | Unity API が `float[]` を要求するため変換コストが発生（v1.3.x で不採用決定済み） |
| Builder パターン (PiperConfig) | Unity の `[Serializable]` + Inspector 編集と相性が悪い（v1.3.x で不採用決定済み） |
| ISynthesisPipeline 抽象 | piper-plus も不採用。YAGNI。ストリーミング対応時に再評価（v1.3.x で不採用決定済み） |
| 言語ヒント API `[sv]` タグ | `defaultLatinLanguage` 切り替えで代替可能（v1.3.x で不採用決定済み） |
| Phoneme ID `int` → `long` 統一 | Sentis が int32 テンソル要求。C# 側で long を使うメリットなし（v1.3.x で不採用決定済み） |
| MultilingualPhonemizer 全同期化（async 完全廃止） | WebGL の `DotNetG2PPhonemizer.InitializeAsync()` が非同期必須。メソッドシグネチャは async 維持が安全 |

## マイグレーションガイド概要

### v1.x → v2.0 で必要な変更

#### 1. PuaTokenMapper 参照の更新

```csharp
// v1.x (static)
var ch = PuaTokenMapper.Register("ch");
var mapped = PuaTokenMapper.MapSequence(tokens);

// v2.0 (instance)
var mapper = new PuaTokenMapper();          // or DI
var ch = mapper.Register("ch");
var mapped = mapper.MapSequence(tokens);
```

#### 2. MultilingualPhonemizer コンストラクタ

```csharp
// v1.x (Obsolete constructor -- 削除)
var mp = new MultilingualPhonemizer(languages, "en", jaPhonemizer, ...);

// v2.0 (Options のみ)
var options = new MultilingualPhonemizerOptions
{
    Languages = languages,
    DefaultLatinLanguage = "en",
    // 個別エンジンプロパティ → ILanguageG2PHandler 登録に移行
};
var mp = new MultilingualPhonemizer(options);
```

#### 3. PhonemeIdMap 型変更

```csharp
// v1.x
Dictionary<string, int> phonemeIdMap = voiceConfig.PhonemeIdMap;
int id = phonemeIdMap["a"];

// v2.0
Dictionary<string, int[]> phonemeIdMap = voiceConfig.PhonemeIdMap;
int[] ids = phonemeIdMap["a"];  // 通常は ids.Length == 1
```

#### 4. ValidatedPiperConfig アクセスパス

```csharp
// v1.x
var backend = validatedConfig.Backend;
var sampleRate = validatedConfig.SampleRate;

// v2.0
var backend = validatedConfig.Inference.Backend;
var sampleRate = validatedConfig.Audio.SampleRate;
```

#### 5. IPhonemizerBackend 廃止

```csharp
// v1.x (テストスタブ)
var options = new MultilingualPhonemizerOptions
{
    EnPhonemizer = mockBackend,  // IPhonemizerBackend
};

// v2.0 (ILanguageG2PHandler)
var options = new MultilingualPhonemizerOptions
{
    Handlers = new Dictionary<string, ILanguageG2PHandler>
    {
        ["en"] = new MockEnglishHandler(),
    },
};
```
