# P2-3: ShortTextMitigatingGenerator 追従

## 1. タスク目的とゴール

`ShortTextMitigatingGenerator` を P2-1/P2-2 で変更された `InferenceOutput` 戻り値型に追従させ、PAD 挿入時に durations 配列から挿入位置のエントリを正しく除去するロジックを実装する。

### なぜ必要か

- P2-1 で `GenerateAudioAsync` の戻り値が `Task<InferenceOutput>` に変更されたため、ShortTextMitigatingGenerator のデコレータロジックを追従させる必要がある
- Strategy A（PAD パディング）で音素 ID 列に PAD を挿入した場合、モデルは挿入された PAD に対応する durations エントリも出力する。タイミング計算の正確性のため、挿入した PAD 位置の durations エントリを除去し、元の音素列に対応する durations のみを返す必要がある
- PAD 未挿入時（`phonemeIds.Length >= 40`）は `InferenceOutput` をそのまま透過させる

### 2.2 GenerateAudioAsync の変更

```csharp
public async Task<InferenceOutput> GenerateAudioAsync(
    int[] phonemeIds,
    int[] prosodyFlat = null,
    float lengthScale = 1.0f,
    float noiseScale = 0.667f,
    float noiseW = 0.8f,
    int speakerId = 0,
    int languageId = 0,
    CancellationToken cancellationToken = default)
{
    if (phonemeIds == null || phonemeIds.Length == 0)
        throw new ArgumentException("phonemeIds must not be null or empty.", nameof(phonemeIds));

    var originalCount = phonemeIds.Length;
    var wasPadded = ShortTextProcessor.NeedsPadding(phonemeIds);

    // PAD 挿入位置を記録（durations 除去に使用）
    int afterBos = 0;
    int beforeEos = 0;
    if (wasPadded)
    {
        var deficit = ShortTextProcessor.MinPhonemeIds - phonemeIds.Length;
        afterBos = deficit / 2;
        beforeEos = deficit - afterBos;

        (phonemeIds, prosodyFlat) = ShortTextProcessor.PadPhonemeIds(
            phonemeIds, prosodyFlat);
    }

    if (originalCount < ShortTextProcessor.MinPhonemeIds)
    {
        (noiseScale, noiseW) = ShortTextProcessor.AdjustScales(
            originalCount, noiseScale, noiseW);
    }

    var output = await _inner.GenerateAudioAsync(
        phonemeIds, prosodyFlat,
        lengthScale, noiseScale, noiseW,
        speakerId, languageId, cancellationToken);

    if (wasPadded)
    {
        output = ProcessPaddedOutput(output, afterBos, beforeEos);
    }

    return output;
}
```

### 2.3 ProcessPaddedOutput メソッド（新規）

PAD 挿入後の推論結果から、Audio のトリムと durations の PAD エントリ除去を一括で行う。

```csharp
/// <summary>
/// PAD 挿入後の推論結果を処理する。
/// Audio は無音トリムを適用し、Durations は PAD 挿入位置のエントリを除去する。
/// 元の InferenceOutput は Dispose される。
/// </summary>
/// <param name="output">PAD 挿入後の推論結果。このメソッド内で Dispose される。</param>
/// <param name="afterBos">BOS 直後に挿入された PAD 数。</param>
/// <param name="beforeEos">EOS 直前に挿入された PAD 数。</param>
/// <returns>処理済みの InferenceOutput。呼び出し元が Dispose する。</returns>
private static InferenceOutput ProcessPaddedOutput(
    InferenceOutput output, int afterBos, int beforeEos)
{
    // --- Audio: 既存の無音トリム ---
    var trimmedAudio = ShortTextProcessor.TrimSilence(output.Audio);
    // TrimSilence は元の Audio を Dispose して新しい NativeArray を返す場合がある。
    // ただし output.Audio と trimmedAudio が同一インスタンスの可能性もある（トリム不要時）。

    // --- Durations: PAD 挿入位置を除去 ---
    NativeArray<float> cleanedDurations = default;
    if (output.HasDurations)
    {
        cleanedDurations = RemovePadDurations(
            output.Durations, afterBos, beforeEos);
        // 元の Durations を Dispose（Audio は TrimSilence が管理済み）
        output.Durations.Dispose();
    }

    // 注意: output.Audio は TrimSilence が管理済み（Dispose 済みまたは trimmedAudio と同一）。
    // output.Durations は上で Dispose 済み。
    // output 自体の Dispose は呼ばない（各フィールドを個別管理したため）。

    return new InferenceOutput(trimmedAudio, cleanedDurations);
}
```

### 2.4 RemovePadDurations メソッド（新規）

```csharp
/// <summary>
/// PAD 挿入位置に対応する durations エントリを除去する。
/// BOS 直後の afterBos 個と EOS 直前の beforeEos 個を除去し、
/// 残りを新しい NativeArray にコピーして返す。
/// </summary>
/// <remarks>
/// PAD 挿入パターン（ShortTextProcessor.PadPhonemeIds と対応）:
/// <code>
/// パディング後: [BOS, PAD*afterBos, body..., PAD*beforeEos, EOS]
/// durations:    [d_BOS, d_PAD*afterBos, d_body..., d_PAD*beforeEos, d_EOS]
/// 除去後:       [d_BOS, d_body..., d_EOS]
/// </code>
/// </remarks>
/// <param name="durations">パディング後のモデル出力 durations。</param>
/// <param name="afterBos">BOS 直後に挿入された PAD 数。</param>
/// <param name="beforeEos">EOS 直前に挿入された PAD 数。</param>
/// <returns>PAD エントリを除去した新しい NativeArray。呼び出し元が Dispose する。</returns>
private static NativeArray<float> RemovePadDurations(
    NativeArray<float> durations, int afterBos, int beforeEos)
{
    var totalPadCount = afterBos + beforeEos;
    if (totalPadCount <= 0 || !durations.IsCreated)
        return durations; // PAD なしまたは未作成: そのまま返す

    var originalLength = durations.Length;
    var newLength = originalLength - totalPadCount;
    if (newLength <= 0)
    {
        // 全て PAD の場合（通常あり得ないが防御）
        return new NativeArray<float>(0, Allocator.Persistent);
    }

    var cleaned = new NativeArray<float>(newLength, Allocator.Persistent);
    try
    {
        // BOS（インデックス 0）をコピー
        cleaned[0] = durations[0];

        // Body（PAD をスキップした本体部分）をコピー
        // パディング後配列: [BOS, PAD*afterBos, body..., PAD*beforeEos, EOS]
        // body の開始: 1 + afterBos
        // body の長さ: originalLength - 1(BOS) - afterBos - beforeEos - 1(EOS)
        var bodyStart = 1 + afterBos;
        var bodyLength = originalLength - 1 - afterBos - beforeEos - 1;
        if (bodyLength > 0)
        {
            NativeArray<float>.Copy(durations, bodyStart, cleaned, 1, bodyLength);
        }

        // EOS（最後の要素）をコピー
        cleaned[newLength - 1] = durations[originalLength - 1];

        return cleaned;
    }
    catch
    {
        if (cleaned.IsCreated)
            cleaned.Dispose();
        throw;
    }
}
```

### 2.5 PAD 除去の正当性検証

`ShortTextProcessor.PadPhonemeIds` のパディングパターン:

```
入力:  [BOS, p1, p2, ..., pN, EOS]  (length < 40)
deficit = 40 - length
afterBos = deficit / 2
beforeEos = deficit - afterBos

出力:  [BOS, PAD*afterBos, p1, p2, ..., pN, PAD*beforeEos, EOS]
```

モデルの durations 出力は入力と同じ長さ:
```
durations: [d_BOS, d_PAD*afterBos, d_p1, d_p2, ..., d_pN, d_PAD*beforeEos, d_EOS]
```

PAD 除去後:
```
durations: [d_BOS, d_p1, d_p2, ..., d_pN, d_EOS]
```

これは元の（パディング前の）音素 ID 列に対応する durations 配列となり、`TimingCalculator` が正しくタイミングを算出できる。

### 2.6 具体例

```
入力 phonemeIds: [1, 3, 4, 2]  (length=4, BOS=1, k=3, o=4, EOS=2)
deficit = 40 - 4 = 36
afterBos = 36 / 2 = 18
beforeEos = 36 - 18 = 18

パディング後: [1, 0*18, 3, 4, 0*18, 2]  (length=40)
durations:    [d0, d1...d18, d19, d20, d21...d38, d39]

除去対象: インデックス 1-18 (afterBos=18個) と 21-38 (beforeEos=18個)
残り:     [d0, d19, d20, d39]  → [d_BOS, d_k, d_o, d_EOS]
```

### 2.7 非パディング時の動作

`phonemeIds.Length >= 40` の場合、`wasPadded == false` となり、`_inner.GenerateAudioAsync` の戻り値 `InferenceOutput` がそのまま返される。durations の操作は一切行わない。

### 2.8 既存の TrimSilence との統合

`ShortTextProcessor.TrimSilence` は `NativeArray<float>` を受け取り、トリムが必要な場合は元の NativeArray を Dispose して新しい NativeArray を返す。トリム不要時は元の NativeArray をそのまま返す。

`ProcessPaddedOutput` では:
1. `TrimSilence(output.Audio)` で Audio をトリム
2. `RemovePadDurations(output.Durations, ...)` で durations の PAD を除去
3. 元の `output.Durations` を Dispose
4. 新しい `InferenceOutput(trimmedAudio, cleanedDurations)` を返す

**注意点**: `TrimSilence` が元の `output.Audio` を Dispose する場合、`output` 自体の `Dispose()` を呼ぶと二重 Dispose になる。そのため `output` の各フィールドを個別に管理し、`output.Dispose()` は呼ばない。

### 2.9 SupportsDurations プロパティの委譲

P2-2 で `IInferenceAudioGenerator` に `SupportsDurations` が追加され、コンパイル通過のため ShortTextMitigatingGenerator にも委譲 `public bool SupportsDurations => _inner.SupportsDurations;` が P2-2 のスコープで追加済み。P2-3 での追加作業は不要。

## 6. 一から作り直すとしたら

### 代替案A: ShortTextProcessor に durations 除去メソッドを追加

```csharp
// ShortTextProcessor に追加
internal static NativeArray<float> RemovePadDurations(
    NativeArray<float> durations, int afterBos, int beforeEos) { ... }
```

**メリット**: パディング関連ロジックが ShortTextProcessor に集約される。テスタビリティが向上（static メソッドを直接テスト可能）。
**デメリット**: ShortTextProcessor が NativeArray<float> に依存する（現在は int[] と NativeArray の2パターン混在を避けている）。

**判定**: `RemovePadDurations` は ShortTextMitigatingGenerator 内の private static メソッドとして配置する。ShortTextProcessor は純粋なユーティリティ（パディング・スケール調整・トリム）を提供し、InferenceOutput 固有のロジックはデコレータ側に配置する設計が適切。将来的に再利用の必要が生じた場合に ShortTextProcessor に移動する。

### 代替案B: InferenceOutput に PAD 除去メソッドを追加

```csharp
internal readonly struct InferenceOutput : IDisposable
{
    // ...
    public InferenceOutput RemovePadDurations(int afterBos, int beforeEos) { ... }
}
```

**メリット**: InferenceOutput が自身の変換を管理する。
**デメリット**: InferenceOutput はデータ保持型であり、変換ロジックを持つべきではない（単一責任原則）。readonly struct にメソッドを追加すると defensive copy 問題のリスクが増加。

**判定**: InferenceOutput はデータ保持に徹し、変換ロジックはデコレータ側に配置する。

### 代替案C: PAD 除去を TimingCalculator 側で実施

```csharp
// TimingCalculator.Calculate() で PAD (ID=0) をスキップする既存ロジックに依存
// → durations の PAD エントリも自動的にスキップされる
```

**メリット**: ShortTextMitigatingGenerator の変更が不要。TimingCalculator が PAD スキップを内包済み。
**デメリット**: durations 配列の長さが phonemeIds（元の長さ）と一致しない。中間層（SplitInferenceOrchestrator、TTSSynthesisOrchestrator）が「パディングされた durations」を透過する必要があり、理解が困難。PAD 除去の責務が適切な層（パディングを行った層）にない。

**判定**: 「パディングを挿入した層が除去も行う」原則に従い、ShortTextMitigatingGenerator で PAD 除去を実施する。これにより下流のコンポーネントはパディングの存在を意識する必要がない。

## 7. 後続タスクへの連絡事項

### T2-1 (ShortTextMitigatingGenerator PAD 除去テスト) への連絡

- `RemovePadDurations` は `private static` メソッドのため、直接テストは不可。`GenerateAudioAsync` の統合テストで検証する
- テストには StubInferenceAudioGenerator（P2-4）を使用し、`DurationsToReturn` に既知の durations を設定して、PAD 除去後の結果を検証する
- テストケース例:
  - 入力 phonemeIds: [1, 3, 4, 2]（length=4）、deficit=36、afterBos=18、beforeEos=18
  - StubInferenceAudioGenerator の DurationsToReturn: 40要素の float[]
  - 期待: 戻り値の InferenceOutput.Durations の長さが 4（BOS + k + o + EOS）
  - 期待: Durations[0] == DurationsToReturn[0]（BOS）
  - 期待: Durations[1] == DurationsToReturn[19]（k、afterBos=18 をスキップ）
  - 期待: Durations[2] == DurationsToReturn[20]（o）
  - 期待: Durations[3] == DurationsToReturn[39]（EOS）

### P3-1 (SplitInferenceOrchestrator durations 結合) への連絡

- ShortTextMitigatingGenerator が IInferenceAudioGenerator をデコレートしている場合、SplitInferenceOrchestrator が受け取る InferenceOutput の durations は PAD 除去済み
- SplitInferenceOrchestrator は句ごとの durations を結合する際、PAD の存在を意識する必要はない

### P3-2 (TTSSynthesisOrchestrator タイミング計算統合) への連絡

- TTSSynthesisOrchestrator が受け取る InferenceOutput の durations は、ShortTextMitigatingGenerator による PAD 除去済みの状態
- durations の長さは元の（パディング前の）phonemeIds.Length と一致する
- TimingCalculator.Calculate() にはパディング前の phonemeIds と除去済み durations を渡すこと
