# T4-2: 統合テスト -- タイミング精度

## 1. タスク目的とゴール

SynthesizeWithTimingAsync() が返すタイミング情報の精度を定量的に検証する。具体的には、最終エントリの EndSeconds と AudioClip.length の差が 50ms 以内であること、および durations の合計値とタイミングの合計値が整合していることを保証する。TimingCalculator の計算精度は T1-1 ~ T1-4 で単体テスト済みだが、本チケットでは PhonemeEncoder -> Generator -> TimingCalculator -> SynthesisWithTimingResult の統合パイプラインを通した場合の精度を検証する。

**ゴール**: 最終 PhonemeTimingEntry の EndSeconds と AudioClip.length の差が 50ms (0.05f) 以内であること、および TotalDurationSeconds と AudioClip.length が一致することを保証する。

## 2. 実装する内容の詳細

### テストファイル
`Assets/uPiper/Tests/Editor/PiperTTSSynthesizeWithTimingTests.cs`（T4-1 で作成するファイルに追加）

### 前提知識

タイミング精度に影響する要因:
1. **frameLength**: `audioSamples / sum(durations)` で計算。StubGenerator が固定長の AudioData を返すため、frameLength は確定的
2. **intersperse PAD の duration**: BOS/EOS/PAD の duration は TimingCalculator でスキップされるが、frameLength 計算には全 duration が使用される
3. **float 精度**: 累積加算による浮動小数点誤差

StubGenerator の AudioDataToReturn の長さと DurationsToReturn の値から、期待される frameLength と各音素のタイミングを事前計算できる:
```
frameLength = audioSamples / sum(durations)
各音素の秒数 = duration * frameLength / sampleRate
```

### 実装するテストメソッド

1. **SynthesizeWithTimingAsync_LastEndSeconds_WithinThresholdOfAudioLength**
   - Arrange:
     - StubGenerator.AudioDataToReturn = float[22050]（1秒分の音声データ）
     - StubGenerator.DurationsToReturn = float[9]（均一値で sum=90.0f）
       - 例: 各要素 10.0f → sum = 90.0f
     - phonemes = ["k", "o", "N"]
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: |result.Timings[^1].EndSeconds - result.AudioClip.length| < 0.05f
   - 最終エントリの EndSeconds と AudioClip.length の差が 50ms 以内

2. **SynthesizeWithTimingAsync_TotalDurationSeconds_MatchesAudioClipLength**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: result.TotalDurationSeconds == result.AudioClip.length (delta=1e-5f)
   - TotalDurationSeconds が AudioClip.length と一致すること

3. **SynthesizeWithTimingAsync_TimingsDurationSum_ConsistentWithAudioLength**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - 計算: sumDuration = Timings の全エントリの DurationSeconds の合計
   - Assert: sumDuration <= result.AudioClip.length
   - 音素タイミングの合計が AudioClip.length を超えないこと（BOS/EOS/PAD の duration 分だけ短い可能性あり）

4. **SynthesizeWithTimingAsync_NonUniformDurations_PrecisionMaintained**
   - Arrange:
     - StubGenerator.AudioDataToReturn = float[44100]（2秒分の音声データ）
     - StubGenerator.DurationsToReturn = float[9]（非均一値）
       - 例: [1.0f, 2.0f, 15.0f, 3.0f, 25.0f, 4.0f, 20.0f, 5.0f, 1.0f]
       - 実音素位置（インデックス 2, 4, 6）: 15.0f, 25.0f, 20.0f
     - phonemes = ["k", "o", "N"]
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: |result.Timings[^1].EndSeconds - result.AudioClip.length| < 0.05f
   - 非均一な durations でも精度が維持されること

5. **SynthesizeWithTimingAsync_LargerPhonemeSet_PrecisionMaintained**
   - Arrange:
     - StubGenerator.AudioDataToReturn = float[44100]（2秒分）
     - phonemes = ["k", "o", "N", "n", "i", "t", "i", "w", "a"]（9音素）
     - PhonemeEncoder エンコード後: BOS + PAD + 9*(phoneme+PAD) + EOS = 21 要素
     - StubGenerator.DurationsToReturn = float[21]（各要素に適切な正の値を設定）
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: result.Timings.Count == 9（実音素数）
   - Assert: |result.Timings[^1].EndSeconds - result.AudioClip.length| < 0.05f
   - より多くの音素での精度維持を検証

### コードパターン例

```csharp
[Test]
public async Task SynthesizeWithTimingAsync_LastEndSeconds_WithinThresholdOfAudioLength()
{
    // Arrange: 1秒分の音声データ、均一な durations
    _stubGenerator.AudioDataToReturn = new float[22050];
    for (var i = 0; i < 22050; i++)
        _stubGenerator.AudioDataToReturn[i] = 0.1f;

    _stubGenerator.DurationsToReturn = new float[9];
    for (var i = 0; i < 9; i++)
        _stubGenerator.DurationsToReturn[i] = 10.0f;

    var config = CreateValidatedConfig(enableSilence: false);
    var orchestrator = new TTSSynthesisOrchestrator(
        _stubGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
        config, _voiceConfig, audioCache: null, puaTokenMapper: _puaTokenMapper);
    var request = new SynthesisRequest(
        new[] { "k", "o", "N" },
        null,
        1.0f, 0.667f, 0.8f,
        0, 0);

    // Act
    var result = await orchestrator.SynthesizeWithTimingAsync(request);

    // Assert
    Assert.IsTrue(result.HasTimings, "タイミング情報が存在すること");
    var lastEntry = result.Timings[result.Timings.Count - 1];
    var diff = Mathf.Abs(lastEntry.EndSeconds - result.AudioClip.length);
    Assert.Less(diff, 0.05f,
        $"最終エントリの EndSeconds ({lastEntry.EndSeconds:F4}) と " +
        $"AudioClip.length ({result.AudioClip.length:F4}) の差が 50ms 以内であること " +
        $"(実差: {diff * 1000:F1}ms)");
}
```

| 1 | AudioClip.length の精度 | Unity の AudioClip.length は samples / frequency で計算されるため、float 精度の範囲内で正確。StubGenerator が返す AudioDataToReturn の長さから確定的に計算できる |

## 6. 一から作り直すとしたら

テスト設計は「最終エントリの EndSeconds と AudioClip.length の 50ms 精度」を主軸に、補助的に TotalDurationSeconds の一致と合計 DurationSeconds の整合性を検証するのが最適。

StubGenerator を使用する場合、AudioDataToReturn の長さと DurationsToReturn の値から frameLength が確定的に計算できるため、理論的には float 精度（< 0.001ms）の誤差しか発生しない。50ms 閾値は実モデル（durations のばらつき・ONNX 推論の非確定性）に対する許容範囲であり、Stub テストでは過剰に緩い閾値だが、将来の実モデルテストとの一貫性のために同一閾値を使用する。

非均一 durations テスト（#4）は、長い音素（母音）と短い音素（PAD/BOS/EOS）の duration 比率が実モデルに近い分布を模倣する。これにより frameLength 計算の精度劣化がないことを検証する。

9 音素テスト（#5）は、PhonemeEncoder の intersperse PAD が増えた場合でも精度が維持されることを確認する。PhonemeIds の長さが増えると DurationsToReturn も長くなり、累積加算の浮動小数点誤差が増える可能性がある。

## 7. 後続タスクへの連絡事項
