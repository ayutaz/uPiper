# T1-3: PUA 逆引きテスト

## 1. タスク目的とゴール

TimingCalculator.Calculate() 内の逆引き辞書構築（BuildReverseIdMap）と PuaTokenMapper.UnmapChar 連携が正しく動作し、PUA 文字が人間可読な多文字トークンに変換されること、未知 ID が "?" にフォールバックすることを検証する。piper-plus TimingWriterTests.CalculateTiming_PuaReverseMapping と同等のカバレッジを実現する。

## 2. 実装する内容の詳細

### テストファイル
`Assets/uPiper/Tests/Editor/AudioGeneration/TimingCalculatorTests.cs`（T1-1 で作成済みのファイルに追加）

### 前提: PuaTokenMapper の役割

TimingCalculator は phonemeIdMap のキーが PUA 単文字の場合、PuaTokenMapper.UnmapChar() で多文字トークンに逆変換する。この連携がテスト対象:

- `\uE000` (ID=17) -> PuaTokenMapper.UnmapChar('\uE000') -> "a:"
- `\uE019` (ID=40) -> PuaTokenMapper.UnmapChar('\uE019') -> "N_m"

### 実装するテストメソッド

全テストメソッドで `TimingCalculator.Calculate(phonemeIds, durations, phonemeIdMap, puaTokenMapper, SampleRate, HopSize)` の形式で呼び出す。PUA 逆引きが必要なテスト（1, 2, 5）では `new PuaTokenMapper()` インスタンスを puaTokenMapper に渡す（固定マッピングのみ使用）。フォールバックテスト（3, 4）では puaTokenMapper=null でも可。

1. **Calculate_PuaReverseMapping_ResolvesToMultiCharToken**
   - 呼び出し: puaTokenMapper = `new PuaTokenMapper()`
   - 入力: phonemeIds = [17, 40], durations = [3.0f, 2.0f]
   - phonemeIdMap に `\uE000`=17, `\uE019`=40 を含む
   - 検証: entries[0].Phoneme == "a:"
   - 検証: entries[1].Phoneme == "N_m"
   - piper-plus CalculateTiming_PuaReverseMapping と完全対応

2. **Calculate_PuaAndRegularMixed_CorrectPhonemes**
   - 呼び出し: puaTokenMapper = `new PuaTokenMapper()`
   - 入力: phonemeIds = [10, 17, 12]（通常 "a" + PUA "a:" + 通常 "k"）
   - 検証: entries[0].Phoneme == "a", entries[1].Phoneme == "a:", entries[2].Phoneme == "k"
   - PUA と通常音素が混在する実運用パターン

3. **Calculate_UnknownId_FallbackToQuestionMark**
   - 呼び出し: puaTokenMapper = null（フォールバック検証のため不要）
   - 入力: phonemeIds = [999], durations = [5.0f]
   - phonemeIdMap に ID=999 が存在しない（ID=999 は printable ASCII 範囲外）
   - 検証: entries[0].Phoneme == "?"
   - piper-plus の UNKNOWN_PHONEME フォールバック動作と一致

4. **Calculate_AsciiPrintableUnknownId_ReturnsAsciiChar**
   - 呼び出し: puaTokenMapper = null
   - 入力: phonemeIds に printable ASCII 範囲（3-126、特殊トークン 0-2 を除く）内で phonemeIdMap に未登録の ID を使用（例: ID=65）
   - 検証: entries[0].Phoneme == "A"（ID=65 は ASCII 'A'）
   - P1-4 の ResolvePhonemeString 仕様: `id is > 2 and < 128` の場合はその ASCII 文字を返す（piper-plus と同一動作）

5. **Calculate_MultiplePuaEntries_AllResolved**
   - 呼び出し: puaTokenMapper = `new PuaTokenMapper()`
   - 日本語 G2P で頻出する PUA マッピングを複数含むケース
   - phonemeIdMap に `\uE000`(a:), `\uE005`(cl), `\uE00E`(ch) を追加
   - 全 PUA エントリが正しく逆変換されることを検証

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

1. **PuaTokenMapper インスタンス管理**: P1-4 の仕様で確定済み。TimingCalculator.Calculate() の第4引数として `PuaTokenMapper puaTokenMapper`（nullable）を受け取る。テストでは `new PuaTokenMapper()` でインスタンス生成（固定マッピングのみ使用）
2. **piper-plus との逆引きロジック差異**: piper-plus は `Mapping.OpenJTalkToPiperMapping.CharToToken` で静的にPUA解決するが、uPiper は `PuaTokenMapper.UnmapChar()` インスタンスメソッドを使用。動的登録された PUA エントリも解決対象となる
3. **phonemeIdMap のキー重複**: 同一 ID に複数のキーがマッピングされている場合の挙動。BuildReverseIdMap で TryAdd を使用するため先勝ちとなるが、テストで明示的に確認すべきか検討

### 5.2 レビューチェックリスト

- [ ] PuaTokenMapper のインスタンス生成がテスト SetUp に含まれているか
- [ ] piper-plus CalculateTiming_PuaReverseMapping の検証項目を網羅しているか
- [ ] 未知 ID フォールバック ("?") のテストがあるか（非 ASCII 範囲: ID >= 128 or ID < 0）
- [ ] printable ASCII 範囲 (3-126) の未知 ID が ASCII 文字を返すテストがあるか（piper-plus ResolvePhonemeString と同一動作）
- [ ] PUA エントリの逆変換テストで使用する PUA コードポイントが FixedPuaMapping の範囲内（0xE000-0xE061）であるか
- [ ] テスト用 phonemeIdMap に PUA エントリが正しく含まれているか（T1-1 の CreateTimingTestPhonemeIdMap を拡張）

## 6. 一から作り直すとしたら

PUA 逆引きのテストは、PuaTokenMapper 単体テスト（PuaTokenMapperTests.cs に既存）と TimingCalculator 内の逆引き統合テストの2層構成が望ましい。本チケットは後者（統合レベル）に焦点を当てる。PuaTokenMapper.UnmapChar() 自体のテストは既存の PuaTokenMapperTests で十分にカバーされているため、ここでは TimingCalculator が PuaTokenMapper を正しく呼び出し、結果を PhonemeTimingEntry.Phoneme に反映しているかを検証する。

## 7. 後続タスクへの連絡事項

- T1-1 の CreateTimingTestPhonemeIdMap() に PUA エントリ（`\uE000`=17, `\uE019`=40）が含まれていることが前提。追加の PUA エントリ（`\uE005`=cl, `\uE00E`=ch）が必要な場合はヘルパーを拡張するか、テストメソッド内でローカルに構築する
- PuaTokenMapper インスタンスのライフサイクル（テストごとに new するか、[SetUp] で共有するか）は T1-1 のセットアップ方針に合わせる
- T1-4 のエッジケーステストでも phonemeIdMap に存在しない ID のテストがあるが、本チケットの未知 ID テストは ResolvePhonemeString のフォールバック動作（"?" / ASCII 文字）に特化。T1-4 は null 入力・境界値・例外系に特化
- 注意: Calculate_AsciiPrintableUnknownId テストのメソッド名が T1-3 では `ReturnsAsciiChar`。P1-4 の ResolvePhonemeString 仕様（`id is > 2 and < 128` で ASCII 文字返却）に基づく
