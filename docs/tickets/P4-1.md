# P4-1: IPiperTTS インターフェース拡張

## 1. タスク目的とゴール

公開インターフェース `IPiperTTS` にタイミング情報付き音声合成メソッド `SynthesizeWithTimingAsync` を追加する。これにより、外部パッケージ利用者が `SynthesisWithTimingResult`（AudioClip + フォネムタイミング情報）を取得可能になる。

**ゴール**: `IPiperTTS.cs` に `SynthesizeWithTimingAsync` メソッド宣言を1件追加し、既存メソッドに一切変更を加えずコンパイルが通ること。

### なぜ必要か

- Phase 1-3 で `SynthesisWithTimingResult`（P1-3）、`TimingCalculator`（P1-4）、`TTSSynthesisOrchestrator.SynthesizeWithTimingAsync`（P3-2, internal）が実装されたが、外部からアクセスする公開 API が未定義
- `IPiperTTS` に宣言を追加することで、DI・テストダブル・将来の実装差し替えが可能になる
- P4-2（`PiperTTS.Inference.cs` 実装）の前提条件として、インターフェース宣言が先に必要

## 2. 実装する内容の詳細

### 2.1 変更対象ファイル

| ファイル | 変更内容 |
|---------|---------|
| `Assets/uPiper/Runtime/Core/IPiperTTS.cs` | `SynthesizeWithTimingAsync` メソッド宣言を追加 |

### 2.2 追加するメソッド宣言

既存の `SynthesizeAsync` 宣言の直後に追加する。

```csharp
/// <summary>
/// SynthesisRequestを直接指定してタイミング情報付き音声を生成する（低レベルAPI）。
/// 音素列は事前に <see cref="PhonemizeAsync"/> または外部G2Pで取得・構築済みであること。
/// </summary>
/// <param name="request">音声合成リクエスト（音素・Prosody・合成パラメータを集約）。</param>
/// <param name="cancellationToken">キャンセルトークン。</param>
/// <returns>
/// タイミング情報付き音声合成結果。
/// <see cref="SynthesisWithTimingResult.HasTimings"/> が false の場合、
/// モデルが durations テンソルを出力しない（非対応モデル）か、
/// 句分割推論が使用されたことを意味する。その場合でも AudioClip は有効。
/// </returns>
/// <exception cref="ObjectDisposedException">インスタンスがDispose済みの場合。</exception>
/// <exception cref="InvalidOperationException">Inferenceが初期化されていない場合。</exception>
Task<SynthesisWithTimingResult> SynthesizeWithTimingAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default);
```

### 2.3 追加位置

既存の `SynthesizeAsync` 宣言と `PhonemizeAsync` 宣言の間に配置する。関連する低レベル API をまとめることで、インターフェース内の論理的なグルーピングを維持する。

```csharp
// 既存（変更なし）
Task<AudioClip> SynthesizeAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default);

// ★ ここに追加
Task<SynthesisWithTimingResult> SynthesizeWithTimingAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default);

// 既存（変更なし）
Task<PhonemizeResult> PhonemizeAsync(
    string text,
    CancellationToken cancellationToken = default);
```

### 2.4 using ディレクティブ

`SynthesisWithTimingResult` は `uPiper.Core.AudioGeneration` 名前空間に定義されている。`IPiperTTS.cs` は既に `using uPiper.Core.AudioGeneration;` を持つため、追加の using は不要。既存の `SynthesizeAsync` が `SynthesisRequest` を修飾なしで参照しているパターンに合わせ、`SynthesisWithTimingResult` も修飾なしで使用する。

### 2.5 設計判断の根拠

| 項目 | 判断 | 理由 |
|------|------|------|
| 既存 `SynthesizeAsync` の変更 | 一切なし | 破壊的変更を避け後方互換を維持。既存ユーザーは AudioClip のみが必要なケースが多い |
| 高レベル API（`GenerateAudioAsync`）への追加 | なし | 要件定義書の方針通り。高レベル API はテキスト入力 → AudioClip の簡潔さを維持し、タイミング情報は低レベル API のみで提供 |
| メソッド名 | `SynthesizeWithTimingAsync` | 既存 `SynthesizeAsync` との対称性。`WithTiming` サフィックスで追加情報を明示 |
| 戻り値型 | `Task<SynthesisWithTimingResult>` | P1-3 で定義済みの公開型。AudioClip + Timings + TotalDurationSeconds + HasTimings を集約。`using uPiper.Core.AudioGeneration;` により修飾なしで参照（既存 `SynthesisRequest` / `PhonemizeResult` と同一パターン） |
| CancellationToken のデフォルト値 | `default` | 既存メソッドと同一パターン |
| XML doc の durations 非対応説明 | `HasTimings == false` のケースを明記 | 利用者が null チェックの必要性を認識できるようにする。`Timings` が null になる2つのケース（モデル非対応・句分割使用時）を列挙。`<see cref="SynthesisWithTimingResult.HasTimings"/>` で参照（修飾なし） |

### 2.6 コーディング規約準拠

- アクセス修飾子: インターフェースメンバーのため省略（C#のデフォルト `public` を適用）
- 既存の `SynthesizeAsync` / `PhonemizeAsync` 宣言と同じフォーマット（インデント、改行位置）を維持
- XML ドキュメントコメント必須（public API のため）
- `<see cref="..."/>` で関連型へのクロスリファレンスを追加

### 2.7 変更しないもの

以下は明示的にスコープ外とし、一切変更しない:

- 既存の `SynthesizeAsync` メソッド宣言
- 既存の `PhonemizeAsync` メソッド宣言
- 既存の `GenerateAudioAsync` 3オーバーロード
- その他すべての既存メソッド・プロパティ・イベント宣言
- using ディレクティブ
- `IPiperTTS.cs` 以外のファイル

## 6. 一から作り直すとしたら

現設計は適切である。再設計するとしても以下の判断は維持する:

- **既存 `SynthesizeAsync` を変更しない**: 破壊的変更を避ける最も重要な制約。戻り値型を `SynthesisWithTimingResult` に変更する案は、全既存ユーザーのコード修正を要求するため不適切
- **低レベル API のみ提供**: 高レベル `GenerateAudioAsync` にタイミング版を追加すると、テキスト → 音素化 → 合成の全パイプラインでタイミングを保持する必要があり、API が複雑化する。低レベル API であれば利用者が明示的にタイミング情報を要求するため、意図が明確
- **メソッド名の対称性**: `SynthesizeAsync` / `SynthesizeWithTimingAsync` のペアは直感的

変更を検討する点:
- `SynthesizeWithTimingAsync` ではなく `SynthesizeDetailedAsync` や `SynthesizeExtendedAsync` も候補だが、`WithTiming` の方が返却情報の内容を正確に表している
