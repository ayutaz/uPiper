# P1-1: PhonemeTimingEntry 定義

## 1. タスク目的とゴール

VITSモデルの `durations` テンソルから算出される音素ごとのタイミング情報を保持するデータ型を定義する。この型はフォネムタイミング機能全体の基盤型であり、TimingCalculator（P1-4）、SynthesisWithTimingResult（P1-3）、AudioSynthesisCache（P3-3）、および公開API（P4-1/P4-2）から参照される。

piper-plus の `TimingWriter.PhonemeTimingEntry`（record型、L37-41）と同等のデータを保持するが、uPiperのコーディング規約に合わせて `public readonly struct` として定義する。

### なぜ必要か

- Live2Dリップシンク、3Dブレンドシェイプ制御、字幕表示など、音素タイミングに依存するユースケースが要件にある
- 後続の全フェーズ（P1-3, P1-4, P3-3, P4-1/P4-2）がこの型を参照するため、最初に定義が必要

## 2. 実装する内容の詳細

### 2.1 新規ファイル

**パス**: `Assets/uPiper/Runtime/Core/AudioGeneration/PhonemeTimingResult.cs`

### 2.2 型定義

```csharp
namespace uPiper.Core.AudioGeneration
{
    /// <summary>
    /// 1音素のタイミング情報。
    /// ONNX モデルの <c>durations</c> 出力テンソルから算出される。
    /// </summary>
    /// <remarks>
    /// piper-plus <c>TimingWriter.PhonemeTimingEntry</c> と同等のデータを保持する。
    /// 時刻は発話先頭からの秒数。ミリ秒変換はユーザー側の責務。
    /// </remarks>
    public readonly struct PhonemeTimingEntry
    {
        /// <summary>人間可読な音素文字列（"a:", "k", "N_m" 等）。</summary>
        public string Phoneme { get; }

        /// <summary>開始時刻（秒）。発話先頭からの経過時間。</summary>
        public float StartSeconds { get; }

        /// <summary>終了時刻（秒）。発話先頭からの経過時間。</summary>
        public float EndSeconds { get; }

        /// <summary>
        /// 持続時間（秒）。<c>EndSeconds - StartSeconds</c> と等価な算出プロパティ。
        /// API利便性のため冗長に保持する。
        /// </summary>
        public float DurationSeconds => EndSeconds - StartSeconds;

        /// <summary>
        /// PhonemeTimingEntry を構築する。
        /// </summary>
        /// <param name="phoneme">人間可読な音素文字列。</param>
        /// <param name="startSeconds">開始時刻（秒）。</param>
        /// <param name="endSeconds">終了時刻（秒）。</param>
        internal PhonemeTimingEntry(string phoneme, float startSeconds, float endSeconds)
        {
            Phoneme = phoneme;
            StartSeconds = startSeconds;
            EndSeconds = endSeconds;
        }
    }
}
```

### 2.3 設計判断の根拠

| 判断 | 理由 |
|------|------|
| `public readonly struct` | `SynthesisRequest` と同じパターン（CLAUDE.md記載）。GCフリー値型、配列格納時にヒープアロケーションなし |
| `internal` コンストラクタ | `PhonemizeResult`（L33）と同じパターン。外部からの不正なインスタンス化を防止。`TimingCalculator`（P1-4）のみが構築する |
| `DurationSeconds` は算出プロパティ | piper-plus は `record` のコンストラクタ引数として duration を受け取るが、readonly struct では冗長なフィールド格納は不要。`EndSeconds - StartSeconds` で一貫性を保証。メモリ使用量も `float` 1個分（4bytes）削減 |
| 単位は秒 | piper-plus `TimingWriter.PhonemeTimingEntry` と同一単位。ミリ秒変換は `* 1000f` で自明であり、ユーザー側の責務 |
| `Phoneme` は `string` | PUA文字ではなく人間可読文字列。`TimingCalculator` が `PuaTokenMapper.UnmapChar` で逆引きして格納する |
| file-scoped namespace 不使用 | 既存コード（`PhonemizeResult.cs`, `SynthesisRequest.cs`）が従来スタイルを使用しているため統一 |
| コンストラクタ引数は3つ（durationなし） | `DurationSeconds` が算出プロパティであるため、コンストラクタに `duration` 引数は不要。piper-plus との差分だが、算出値をコンストラクタで受け取ると不整合リスクがある |

### 2.4 piper-plus との対応

| piper-plus (`TimingWriter.PhonemeTimingEntry`) | uPiper (`PhonemeTimingEntry`) | 差分 |
|---|---|---|
| `record` (C# 10, 参照型) | `readonly struct` (値型) | GCフリー、Unityパフォーマンス最適化 |
| コンストラクタ4引数（duration含む） | コンストラクタ3引数（duration算出） | 不整合リスク排除 |
| `public` コンストラクタ | `internal` コンストラクタ | 既存パターン踏襲 |
| file-scoped namespace | 従来namespace | 既存コードスタイル統一 |

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

| # | 懸念 | 対策 |
|---|------|------|
| 1 | `DurationSeconds` が算出プロパティのため、`float` 減算の浮動小数点誤差が蓄積する可能性 | TimingCalculator（P1-4）が `currentTime += frameDuration * frameLength` で累積計算するため、start/end 自体に既に丸め誤差が含まれる。算出プロパティの追加誤差は無視できるレベル。piper-plus でも同じ計算方式 |
| 2 | `internal` コンストラクタにしたことで、ユーザーがカスタム `PhonemeTimingEntry` を構築できない | 要件上、タイミング情報はモデル出力からのみ生成される。ユーザー構築の必要性がない。将来的に必要になった場合は `public` ファクトリメソッドを追加 |
| 3 | `Phoneme` プロパティが `null` を許容する | readonly struct のデフォルト値は全フィールドがゼロ初期化されるため `Phoneme == null` は避けられない。TimingCalculator（P1-4）が `"?"` フォールバックで null 生成を回避。ドキュメントで明記 |

### 5.2 レビューチェックリスト

- [ ] `public readonly struct` であること（`class` や `struct` のみでないこと）
- [ ] `internal` コンストラクタであること（`public` でないこと）
- [ ] `DurationSeconds` が算出プロパティ（`=>` 式）であること（バッキングフィールドがないこと）
- [ ] コンストラクタ引数に `duration` パラメータがないこと
- [ ] 従来スタイル namespace（file-scoped でないこと）
- [ ] フィールド命名が PascalCase（`Phoneme`, `StartSeconds`, `EndSeconds`, `DurationSeconds`）
- [ ] XMLドキュメントコメントが全 public メンバーに付与されていること
- [ ] `using` 文が必要最小限であること（`System` のみ、Unity依存なし）
- [ ] ファイル末尾に余分な空行がないこと（dotnet-format FINALNEWLINE対応）

## 6. 一から作り直すとしたら

### 代替案A: C# record struct

```csharp
public readonly record struct PhonemeTimingEntry(
    string Phoneme, float StartSeconds, float EndSeconds)
{
    public float DurationSeconds => EndSeconds - StartSeconds;
}
```

**メリット**: `ToString()`, `Equals()`, `GetHashCode()` が自動生成。コード量が最小。
**デメリット**: `public` コンストラクタが強制される（record struct はコンストラクタを `internal` にできない）。既存パターン（PhonemizeResult, SynthesisRequest）との一貫性が失われる。

**判定**: 既存パターンとの一貫性を優先し、通常の `readonly struct` + `internal` コンストラクタを採用。

### 代替案B: DurationSeconds をバッキングフィールドに変更

```csharp
public float DurationSeconds { get; }  // コンストラクタで格納

internal PhonemeTimingEntry(string phoneme, float startSeconds, float endSeconds, float durationSeconds)
{
    // ...
    DurationSeconds = durationSeconds;
}
```

**メリット**: piper-plus と完全互換。プロパティアクセスのたびに減算しない（ただし最適化で inline 化されるため実質差なし）。
**デメリット**: `EndSeconds - StartSeconds != DurationSeconds` になるリスク（呼び出し元が不正な値を渡した場合）。4bytes のメモリ増加。

**判定**: 不整合リスク排除のため算出プロパティを採用。JIT が `EndSeconds - StartSeconds` を inline 化するためパフォーマンス差なし。

### 代替案C: 秒ではなくフレーム数を保持

```csharp
public readonly struct PhonemeTimingEntry
{
    public int StartFrame { get; }
    public int EndFrame { get; }
    public float StartSeconds => StartFrame * _frameLength;
    // ...
}
```

**メリット**: 整数値で精度損失なし。ユーザーがフレーム単位でも秒単位でも取得可能。
**デメリット**: `_frameLength` をインスタンスに持たせる必要がありメモリ増加。APIが複雑化。piper-plus との互換性喪失。

**判定**: piper-plus 互換性とAPIシンプルさを優先し、秒単位を採用。

## 7. 後続タスクへの連絡事項

### P1-3 (SynthesisWithTimingResult) への連絡

- `PhonemeTimingEntry` は値型のため、`IReadOnlyList<PhonemeTimingEntry>` で公開する場合は `PhonemeTimingEntry[]` を `Array.AsReadOnly()` でラップするか、配列を直接公開して防御的コピーする（PhonemizeResult パターン）
- `default(PhonemeTimingEntry)` は `Phoneme == null` となる。null チェックが必要な場合は配列の null 性（タイミング非対応）で判断すること

### P1-4 (TimingCalculator) への連絡

- コンストラクタは `internal` のため、同一アセンブリ（`uPiper.Runtime`）内からのみ構築可能
- コンストラクタシグネチャ: `PhonemeTimingEntry(string phoneme, float startSeconds, float endSeconds)`
- `DurationSeconds` はコンストラクタ引数にない。`EndSeconds - StartSeconds` で自動算出される
- piper-plus `TimingWriter.CalculateTiming` L112 の `new PhonemeTimingEntry(phonemeStr, startTime, endTime, endTime - startTime)` は、uPiperでは `new PhonemeTimingEntry(phonemeStr, startTime, endTime)` に変更

### P3-3 (AudioSynthesisCache) への連絡

- `PhonemeTimingEntry` のメモリサイズ: string参照(8bytes) + float x 2(8bytes) = 16bytes（64bit環境）。`DurationSeconds` はバッキングフィールドなし
- 100音素のタイミング配列: ~1.6KB。キャッシュエントリのメモリ計算に加算すること

### T1-5 (PhonemeTimingEntry 構造テスト) への連絡

- テストアセンブリは `InternalsVisibleTo` で `internal` コンストラクタにアクセス可能（既存 `SynthesisRequestTests.cs` L201 で確認済み）
- `default(PhonemeTimingEntry)` のテストを含めること（Phoneme == null, 各float == 0）
