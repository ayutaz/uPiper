# T2-2: InferenceOutput Dispose テスト

## 1. タスク目的とゴール

InferenceOutput（internal readonly struct : IDisposable）の Dispose パターンが全ての NativeArray 状態の組み合わせで安全に動作することを検証する。P1-2 で定義された InferenceOutput の Audio/Durations 両方の NativeArray が正しく破棄されること、未作成（default）の NativeArray に対する Dispose が例外を投げないこと、`default(InferenceOutput).Dispose()` が安全であることを保証する。

**ゴール**: NativeArray のライフサイクル管理が InferenceOutput.Dispose() 経由で一元化されており、いかなる状態の組み合わせでも例外なく安全に動作することを保証する。

### テスト対象の InferenceOutput 仕様（P1-2 より）

```csharp
internal readonly struct InferenceOutput : IDisposable
{
    public NativeArray<float> Audio { get; }
    public NativeArray<float> Durations { get; }
    public bool HasDurations => Durations.IsCreated;

    internal InferenceOutput(NativeArray<float> audio, NativeArray<float> durations);

    public void Dispose()
    {
        if (Audio.IsCreated) Audio.Dispose();
        if (Durations.IsCreated) Durations.Dispose();
    }
}
```

### コードパターン例

```csharp
[Test]
public void Dispose_BothArraysCreated_DisposesAudioAndDurations()
{
    var audio = new NativeArray<float>(100, Allocator.Temp);
    var durations = new NativeArray<float>(10, Allocator.Temp);
    try
    {
        var output = new InferenceOutput(audio, durations);

        // Act
        output.Dispose();

        // Assert: Dispose 後は IsCreated == false
        // NativeArray は Dispose 後に IsCreated が false を返す
        Assert.IsFalse(audio.IsCreated,
            "Audio NativeArray should be disposed");
        Assert.IsFalse(durations.IsCreated,
            "Durations NativeArray should be disposed");
    }
    catch
    {
        // テスト失敗時のリーク防止
        if (audio.IsCreated) audio.Dispose();
        if (durations.IsCreated) durations.Dispose();
        throw;
    }
}

[Test]
public void Dispose_DefaultInferenceOutput_NoException()
{
    Assert.DoesNotThrow(() => default(InferenceOutput).Dispose());
}
```

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

| # | 懸念 | 対策 |
|---|------|------|
| 1 | `readonly struct` + `IDisposable` の defensive copy 問題 | テスト内では `var output = new InferenceOutput(...)` のローカル変数に格納し、`readonly` フィールドに格納しない。テスト自体は正常に動作するが、テストコメントで defensive copy の注意を記載する |
| 2 | NativeArray の Dispose 後に `IsCreated` が false を返すかの Unity バージョン依存 | Unity 2022+ では Dispose 後 `IsCreated == false` が保証される。テスト実行環境の Unity バージョンを確認すること。代替として `Assert.DoesNotThrow` で Dispose の安全性のみ検証するアプローチも可 |
| 3 | `Allocator.Temp` vs `Allocator.Persistent` | テストでは `Allocator.Temp` を使用する（テストフレーム内で完結するため）。実運用では `Allocator.Persistent` が使われるが、Dispose の動作は同一。ShortTextProcessorTests.cs で `Allocator.Temp` と `Allocator.Persistent` の両方が使われているパターンに準拠 |
| 4 | `internal` コンストラクタへのアクセス | テストアセンブリは `InternalsVisibleTo` で `internal` にアクセス可能。`uPiper.Runtime.asmdef` の `internalsVisibleTo` に `uPiper.Tests.Editor` が含まれていることを確認済み（P1-2 §5.1 #4） |
| 5 | NativeArray のテスト失敗時リーク | 全テストで try/catch パターンを使用し、テスト失敗時も NativeArray を確実に Dispose する。既存の ShortTextProcessorTests.cs（TrimSilence テスト）のパターンに準拠 |

## 6. 一から作り直すとしたら

InferenceOutput は `readonly struct : IDisposable` という比較的シンプルな型であり、テスト設計は NativeArray の全状態の組み合わせ（2x2=4 パターン）を網羅する方針が最適。テストの主眼は「Dispose が安全か」であり、NativeArray の IsCreated チェックが正しく機能しているかの検証に集中する。

代替として `Assert.DoesNotThrow` のみで検証する軽量アプローチも考えられるが、Dispose 後の `IsCreated == false` を確認するアプローチの方が NativeArray の実際のリソース解放を検証できるため、より堅牢。ただし Unity バージョンによって Dispose 後の IsCreated の挙動が異なる可能性があるため、`Assert.DoesNotThrow` もバックアップとして含める設計にしている。

HasDurations プロパティのテストは Dispose テストとは独立したカテゴリだが、同じファイルに含めることで InferenceOutput の全機能を1ファイルで見渡せるようにしている。
