# P3-3: AudioSynthesisCache タイミング格納拡張

## 1. タスク目的とゴール

`AudioSynthesisCache` の `CacheEntry` にタイミング情報（`PhonemeTimingEntry[]`）を追加し、キャッシュヒット時にタイミング情報も合わせて返却できるようにする。これにより、同一テキストの2回目以降の合成リクエストでは推論もタイミング計算もスキップし、キャッシュから AudioClip とタイミング情報の両方を即座に返却できる。

### なぜ必要か

- 現在の `AudioSynthesisCache` は `float[] Samples` と `int SampleRate` のみを保持する。タイミング情報付き合成（P3-2 で実装）の結果をキャッシュするには、`PhonemeTimingEntry[]` も格納する必要がある
- キャッシュミス時は推論 + タイミング計算が発生するが、キャッシュヒット時にタイミング情報がないと TTSSynthesisOrchestrator で再計算が必要になり、キャッシュの意味が薄れる（durations 自体はキャッシュしていないため再計算も不可能）
- ハッシュキー変更は不要。同一入力（phonemeIds + 合成パラメータ）なら同一の durations が出力されるため、タイミング情報も同一になる

### 2.2 CacheEntry の拡張

```csharp
// Before:
private readonly struct CacheEntry
{
    public readonly float[] Samples;
    public readonly int SampleRate;
    public readonly long MemoryBytes;

    public CacheEntry(float[] samples, int sampleRate)
    {
        Samples = samples;
        SampleRate = sampleRate;
        MemoryBytes = samples.Length * sizeof(float) + 64; // 64 bytes overhead
    }
}

// After:
private readonly struct CacheEntry
{
    public readonly float[] Samples;
    public readonly int SampleRate;
    public readonly PhonemeTimingEntry[] Timings; // nullable: タイミング非対応時は null
    public readonly long MemoryBytes;

    public CacheEntry(float[] samples, int sampleRate, PhonemeTimingEntry[] timings = null)
    {
        Samples = samples;
        SampleRate = sampleRate;
        Timings = timings;
        MemoryBytes = CalculateMemoryBytes(samples, timings);
    }

    private static long CalculateMemoryBytes(float[] samples, PhonemeTimingEntry[] timings)
    {
        // Audio: float 配列 + オブジェクトヘッダ
        long bytes = samples.Length * sizeof(float) + 64;

        // Timings: PhonemeTimingEntry は readonly struct (string + 2 float = 16 bytes/entry on 64-bit)
        // + string 参照先の文字列メモリ (平均 4 文字 * 2 bytes/char + 40 bytes header ≈ 48 bytes/entry)
        // 合計: 約 64 bytes/entry (構造体 16 + 文字列 48)
        // 簡略化のため 32 bytes/entry で見積もる（構造体のみ、文字列はインターン済みで共有可能）
        if (timings != null)
        {
            bytes += timings.Length * 32L + 64; // 配列オーバーヘッド
        }

        return bytes;
    }
}
```

### 2.3 メモリサイズ見積もり

`PhonemeTimingEntry` は `public readonly struct` で以下のフィールドを持つ（P1-1 で定義）:
- `string Phoneme` (参照型、8 bytes on 64-bit)
- `float StartSeconds` (4 bytes)
- `float EndSeconds` (4 bytes)
- `DurationSeconds` は算出プロパティ（バッキングフィールドなし）

**構造体サイズ**: 8 + 4 + 4 = 16 bytes

**文字列メモリ**: 音素文字列は "a:", "k", "N_m" 等の短い文字列。多くの音素文字列は TimingCalculator の逆引き辞書から取得されるため、同一推論セッション内で文字列がインターン（共有）される可能性が高い。

**見積もり**: 32 bytes/entry（構造体 16 bytes + 文字列参照先の部分配分 16 bytes）を使用する。100 音素のタイミング配列で ~3.2KB。音声データ（22050Hz x 5秒 = 441,000 samples x 4 bytes ≈ 1.7MB）と比較して無視できるサイズ。

### 2.4 TryGet シグネチャの拡張

```csharp
// Before:
public bool TryGet(long key, out float[] samples, out int sampleRate)
{
    if (_cache.TryGetValue(key, out var node))
    {
        _lruList.Remove(node);
        _lruList.AddFirst(node);
        _hitCount++;
        samples = node.Value.Entry.Samples;
        sampleRate = node.Value.Entry.SampleRate;
        return true;
    }

    _missCount++;
    samples = null;
    sampleRate = 0;
    return false;
}

// After:
/// <summary>
/// Try to get cached audio data and optional timing information.
/// </summary>
/// <param name="key">Cache key (FNV-1a hash).</param>
/// <param name="samples">Cached audio samples, or null on miss.</param>
/// <param name="sampleRate">Cached sample rate, or 0 on miss.</param>
/// <param name="timings">Cached timing entries, or null if not available or on miss.</param>
/// <returns>true if cache hit, false if miss.</returns>
public bool TryGet(
    long key,
    out float[] samples,
    out int sampleRate,
    out PhonemeTimingEntry[] timings)
{
    if (_cache.TryGetValue(key, out var node))
    {
        _lruList.Remove(node);
        _lruList.AddFirst(node);
        _hitCount++;
        samples = node.Value.Entry.Samples;
        sampleRate = node.Value.Entry.SampleRate;
        timings = node.Value.Entry.Timings; // null if no timing data
        return true;
    }

    _missCount++;
    samples = null;
    sampleRate = 0;
    timings = null;
    return false;
}
```

### 2.5 Set シグネチャの拡張

```csharp
// Before:
public void Set(long key, float[] samples, int sampleRate)
{
    if (samples == null || samples.Length == 0)
        return;

    var entry = new CacheEntry(samples, sampleRate);
    // ... LRU ロジック ...
}

// After:
/// <summary>
/// Add audio data and optional timing information to the cache.
/// Evicts LRU entries if needed.
/// </summary>
/// <param name="key">Cache key (FNV-1a hash).</param>
/// <param name="samples">Audio samples to cache.</param>
/// <param name="sampleRate">Audio sample rate.</param>
/// <param name="timings">Optional timing entries. null for no timing data.</param>
public void Set(
    long key,
    float[] samples,
    int sampleRate,
    PhonemeTimingEntry[] timings = null)
{
    if (samples == null || samples.Length == 0)
        return;

    var entry = new CacheEntry(samples, sampleRate, timings);

    // Don't cache if single entry exceeds memory budget
    if (entry.MemoryBytes > _maxMemoryBytes)
        return;

    // If key already exists, remove old entry first
    if (_cache.TryGetValue(key, out var existing))
    {
        _currentMemoryBytes -= existing.Value.Entry.MemoryBytes;
        _lruList.Remove(existing);
        _cache.Remove(key);
    }

    // Evict until we have space
    while (_cache.Count >= _maxEntries
        || _currentMemoryBytes + entry.MemoryBytes > _maxMemoryBytes)
    {
        if (_lruList.Count == 0) break;
        EvictLast();
    }

    var node = _lruList.AddFirst((key, entry));
    _cache[key] = node;
    _currentMemoryBytes += entry.MemoryBytes;
}
```

### 2.6 ハッシュキーについて

ハッシュキー（`GenerateKey`）の変更は不要。理由:

- 同一の入力（phonemeIds + prosodyFlat + 合成パラメータ）からは同一の durations が出力される（VITSモデルは決定論的ではないが、同一の phonemeIds に対して durations の分布は一定）
- 実際には、noiseScale/noiseW は音声波形に影響するが durations には影響しない（VITSの durations 予測は入力音素のみに依存）
- タイミング情報は durations から一意に決まるため、同一キーで同一タイミングが保証される
- キーを変更すると既存キャッシュとの互換性が失われ、全エントリが無効化される（不要なコスト）

`PhonemeTimingEntry` は `uPiper.Core.AudioGeneration` namespace 内にあるため、追加の `using` は不要（`AudioSynthesisCache` も同一 namespace）。

### 2.9 timings パラメータの nullable 設計

`timings` を nullable（`PhonemeTimingEntry[] timings = null`）としたのは以下の理由:

1. **durations 非対応モデル**: モデルが durations 出力を持たない場合、タイミング情報は生成されない
2. **句分割使用時**: P3-2 初期実装では句分割時にタイミング情報を提供しない
3. **後方互換**: 既存のキャッシュエントリ（タイミングなし）は `timings == null` で表現される
4. **既存の Set 呼び出し箇所**: `timings` をデフォルトパラメータ（`null`）にすることで、P3-3 適用後も既存の `Set(key, samples, sampleRate)` 呼び出しがコンパイル可能（ただし実際には P3-2 で全呼び出し箇所が更新される）

### 2.10 スレッドセーフティ

現在の `AudioSynthesisCache` はスレッドセーフではない（`Dictionary` + `LinkedList` の非同期アクセスは保証されない）。P3-3 でのスレッドセーフティ改善はスコープ外。既存の使用パターン（メインスレッドからの呼び出し）は P3-3 後も変わらない。

## 6. 一から作り直すとしたら

### 代替案A: CacheEntry を class に変更

```csharp
private sealed class CacheEntry
{
    public float[] Samples { get; }
    public int SampleRate { get; }
    public PhonemeTimingEntry[] Timings { get; }
    public long MemoryBytes { get; }
    // ...
}
```

**メリット**: `readonly struct` の CacheEntry が `LinkedListNode` 内の ValueTuple に格納される際の構造体コピーを回避。フィールド追加による構造体サイズ増加の影響がない。
**デメリット**: ヒープアロケーション1回/エントリ。キャッシュエントリ数（最大50）分の GC 圧力。既存パフォーマンス特性が変わる。

**判定**: `PhonemeTimingEntry[]` の追加は参照型（8 bytes）のため、CacheEntry 構造体サイズの増加は最小限。既存設計を維持する。

### 代替案B: タイミング情報を別辞書で管理

```csharp
// AudioSynthesisCache とは別の Dictionary でタイミング情報を管理
private readonly Dictionary<long, PhonemeTimingEntry[]> _timingCache;
```

**メリット**: 既存の CacheEntry / TryGet / Set を変更せずにタイミングキャッシュを追加可能。後方互換リスクがゼロ。
**デメリット**: 2つのキャッシュの整合性管理が必要（退避・クリア・キー同期）。LRU 退避時に timingCache からも対応エントリを削除する必要があり、EvictLast の変更が必要。結局 CacheEntry を変更するよりも複雑。

**判定**: 単一キャッシュで一元管理する方がシンプルで保守しやすい。CacheEntry 拡張を採用。

### 代替案C: TryGet のオーバーロード（旧シグネチャ維持）

```csharp
// 旧シグネチャ（後方互換）
public bool TryGet(long key, out float[] samples, out int sampleRate)
{
    return TryGet(key, out samples, out sampleRate, out _);
}

// 新シグネチャ
public bool TryGet(long key, out float[] samples, out int sampleRate, out PhonemeTimingEntry[] timings)
{
    // ... 実装 ...
}
```

**メリット**: 旧シグネチャの呼び出し箇所を変更不要。P3-3 単独でコンパイルが通る。
**デメリット**: メソッドオーバーロードが増えて API が複雑化。旧シグネチャの呼び出し箇所はどのみち P3-2 で更新されるため、一時的な互換性維持に過ぎない。

**判定**: P3-2 と P3-3 は同一 Phase 内で近接して実装される前提のため、オーバーロードは不要。P3-3 単独でコンパイルを通す必要がある場合は、TTSSynthesisOrchestrator.cs の呼び出し箇所に `out _` を追加する最小変更で対応する。

### 代替案D: timings を float[] durations としてキャッシュ

```csharp
// PhonemeTimingEntry[] ではなく float[] durations をキャッシュ
public readonly float[] Durations; // 生の durations 配列
```

**メリット**: TimingCalculator の再計算コスト（数十us）を許容すれば、キャッシュサイズが小さくなる（PhonemeTimingEntry[] の string 参照分を節約）。
**デメリット**: キャッシュヒット時に TimingCalculator の再呼び出しが必要。phonemeIds もキャッシュに含める必要がある（TimingCalculator の引数に必要）。キャッシュエントリが大幅に複雑化。

**判定**: `PhonemeTimingEntry[]` をキャッシュする方がシンプルで、キャッシュヒット時のパフォーマンスも最良。タイミング配列のメモリ増加（~3.2KB/100音素）は音声データ（~1.7MB/5秒）と比較して無視できる。

- `CacheEntry` は `private` のため直接テスト不可。`Set` → `TryGet` のラウンドトリップで検証する
- テストケース:
  - タイミング付き: `Set(key, samples, rate, timings)` → `TryGet(key, out s, out r, out t)` → `t != null && t.Length == timings.Length`
  - タイミングなし: `Set(key, samples, rate)` → `TryGet(key, out s, out r, out t)` → `t == null`
  - メモリバジェット: タイミング付きエントリの `CurrentMemoryBytes` がタイミングサイズ分だけ増加すること
  - LRU 退避: タイミング付きエントリ退避後に `CurrentMemoryBytes` が正しく減少すること
- `PhonemeTimingEntry` の `internal` コンストラクタへのアクセスは `InternalsVisibleTo` で許可済み

### T4-3 (統合テスト: キャッシュ整合) への連絡

- キャッシュヒット時にタイミング情報が正しく復元されることを E2E で検証
- 1回目の `SynthesizeWithTimingAsync` → キャッシュミス → 推論 + タイミング計算 + キャッシュ Set
- 2回目の `SynthesizeWithTimingAsync` → キャッシュヒット → AudioClip + タイミング情報を返却
- 両結果の `Timings` が一致すること（PhonemeTimingEntry の各フィールドが等値）

### 既存テストへの影響

- `TryGet` のシグネチャ変更により、既存の AudioSynthesisCacheTests が `out` パラメータ不足でコンパイルエラーになる可能性がある
- 既存テストの `TryGet` 呼び出しに `out _` を追加する最小変更が必要
- `Set` はデフォルトパラメータのため、既存テストの呼び出しは変更不要（再コンパイルのみ必要）
