# T1-2: 特殊トークンスキップテスト

## 1. タスク目的とゴール

TimingCalculator.Calculate() が PAD(ID=0)、BOS(ID=1)、EOS(ID=2) の特殊トークンに対してタイミングエントリを生成せず、時間カーソルのみ正しく進行させることを検証する。piper-plus TimingWriterTests.CalculateTiming_SkipsSpecialTokens と同等のカバレッジを NUnit で実現する。

## 2. 実装する内容の詳細

### テストファイル
`Assets/uPiper/Tests/Editor/AudioGeneration/TimingCalculatorTests.cs`（T1-1 で作成済みのファイルに追加）

### 実装するテストメソッド

全テストメソッドで `TimingCalculator.Calculate(phonemeIds, durations, phonemeIdMap, puaTokenMapper: null, SampleRate, HopSize)` の形式で呼び出す（特殊トークンスキップの検証に PUA 逆引きは不要なため puaTokenMapper=null）。

1. **Calculate_SkipsSpecialTokens_PadBosEos**
   - 入力: phonemeIds = [0, 1, 10, 2], durations = [2.0f, 3.0f, 4.0f, 1.0f]
   - 検証: entries.Length == 1（"a" のみ）
   - 検証: entries[0].Phoneme == "a"
   - 検証: entries[0].StartSeconds == (2+3) * frameLength（PAD+BOS のフレーム分だけ時間が進んでいる）
   - 検証: entries[0].EndSeconds == (2+3+4) * frameLength
   - piper-plus の CalculateTiming_SkipsSpecialTokens と完全対応

2. **Calculate_SkipsPadOnly_TimeAdvances**
   - 入力: phonemeIds = [10, 0, 12], durations = [5.0f, 3.0f, 4.0f]
   - 検証: entries.Length == 2（"a" と "k"）
   - 検証: entries[0].EndSeconds != entries[1].StartSeconds ではなく、PAD の 3 フレーム分のギャップがある
   - entries[1].StartSeconds == (5+3) * frameLength

3. **Calculate_SkipsBosOnly_TimeAdvances**
   - 入力: phonemeIds = [1, 10], durations = [5.0f, 8.0f]
   - 検証: entries.Length == 1
   - 検証: entries[0].StartSeconds == 5 * frameLength（BOS 分だけオフセット）

4. **Calculate_SkipsEosOnly_TimeAdvances**
   - 入力: phonemeIds = [10, 2], durations = [8.0f, 3.0f]
   - 検証: entries.Length == 1
   - 検証: entries[0].EndSeconds == 8 * frameLength（EOS はエントリ生成しないが、計算は完了）

5. **Calculate_AllSpecialTokens_ReturnsEmpty**
   - 入力: phonemeIds = [0, 1, 2], durations = [1.0f, 2.0f, 3.0f]
   - 検証: entries.Length == 0（全て特殊トークンなのでエントリなし）

6. **Calculate_MultiplePads_CorrectTimeGaps**
   - 入力: phonemeIds = [1, 0, 10, 0, 12, 0, 2]（多言語モデルの intersperse PAD パターン）
   - durations = [2.0f, 1.0f, 5.0f, 1.0f, 3.0f, 1.0f, 1.0f]
   - 検証: entries.Length == 2
   - 検証: PAD のフレーム分が時間に正しく反映されていること
   - E8（Intersperse PAD 付き多言語モデル）のカバレッジ

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

1. **特殊トークンID のハードコード**: PAD=0, BOS=1, EOS=2 は PhonemeEncoder の規約に依存。TimingCalculator 内で定数化されているか、テスト側でも同じ定数を参照しているか確認
2. **Intersperse PAD のテストパターン**: PhonemeEncoder が BOS 後にも PAD を挿入するパターン（`[^, _, p1, _, p2, _, $]`）を正しく再現しているか
3. **時間の不連続性**: 特殊トークンスキップ後の音素の startSeconds が前の通常音素の endSeconds と一致しない場合がある（PAD 分のギャップ）。これが仕様通りであることをテストで明示

### 5.2 レビューチェックリスト

- [ ] 3種の特殊トークン（PAD/BOS/EOS）それぞれの個別テストがあるか
- [ ] 時間カーソル進行の検証が全テストに含まれているか
- [ ] piper-plus CalculateTiming_SkipsSpecialTokens の検証を網羅しているか
- [ ] 多言語モデルの Intersperse PAD パターンがカバーされているか (E8)
- [ ] Assert の delta 値が T1-1 と統一されているか（1e-5f）

## 6. 一から作り直すとしたら

特殊トークンのスキップは `id is 0 or 1 or 2` というパターンマッチで判定されるため、テスト設計もこの3値の組み合わせを網羅するアプローチが最適。個別テスト + 組み合わせテスト + 全特殊テストの3段階構成は変更不要。Intersperse PAD パターンのテスト（Calculate_MultiplePads_CorrectTimeGaps）は実運用シナリオに最も近いため優先度が高い。
