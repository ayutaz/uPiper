# T1-5: PhonemeTimingEntry 構造テスト

## 1. タスク目的とゴール

PhonemeTimingEntry（readonly struct）の構造的な正確性を検証する。値の一致、DurationSeconds == EndSeconds - StartSeconds の不変条件、readonly struct としてのイミュータビリティ、デフォルト値の安全性を保証する。

## 2. 実装する内容の詳細

### テストファイル
`Assets/uPiper/Tests/Editor/AudioGeneration/TimingCalculatorTests.cs`（T1-1 で作成済みのファイルに追加）

### テスト対象

```csharp
public readonly struct PhonemeTimingEntry
{
    public string Phoneme { get; }
    public float StartSeconds { get; }
    public float EndSeconds { get; }
    public float DurationSeconds => EndSeconds - StartSeconds; // 算出プロパティ（バッキングフィールドなし）

    internal PhonemeTimingEntry(string phoneme, float startSeconds, float endSeconds)
    // 3引数コンストラクタ（durationは算出なので引数に含めない）
}
```

### 実装するテストメソッド

1. **PhonemeTimingEntry_DurationEqualsEndMinusStart**
   - PhonemeTimingEntry を internal コンストラクタ（3引数: phoneme, startSeconds, endSeconds）で直接構築
   - 検証: DurationSeconds == EndSeconds - StartSeconds（delta=1e-7f）
   - DurationSeconds は算出プロパティ（`=> EndSeconds - StartSeconds`）なので、バッキングフィールドとの不整合リスクがないことを確認
   - 複数の値パターンで検証:
     - (start=0, end=0.5) -> duration=0.5
     - (start=1.234, end=2.567) -> duration=1.333
     - (start=0, end=0) -> duration=0

2. **PhonemeTimingEntry_ConstructedValues_AreCorrect**
   - 3引数コンストラクタで構築し、各プロパティが設定値と一致することを検証
   - Phoneme, StartSeconds, EndSeconds の3プロパティが引数と一致、DurationSeconds が EndSeconds - StartSeconds と一致

3. **PhonemeTimingEntry_Default_HasNullPhonemeAndZeroTimes**
   - `default(PhonemeTimingEntry)` のデフォルト値を検証
   - 検証: Phoneme == null, StartSeconds == 0, EndSeconds == 0, DurationSeconds == 0

4. **PhonemeTimingEntry_EqualityByValue**
   - 同一値で構築した2つの PhonemeTimingEntry が値として等しいことを検証
   - readonly struct のため値型比較が行われる
   - Equals() と == 演算子（実装されている場合）の両方を検証

5. **PhonemeTimingEntry_DifferentValues_NotEqual**
   - 異なる値で構築した2つの PhonemeTimingEntry が等しくないことを検証
   - Phoneme が異なる場合、StartSeconds が異なる場合、それぞれテスト

6. **PhonemeTimingEntry_ZeroDuration_IsValid**
   - start == end の場合（duration=0）が有効であることを検証
   - E3（全ゼロ durations）シナリオで生成されるエントリの妥当性

7. **PhonemeTimingEntry_DurationConsistency_AfterTimingCalculation**
   - TimingCalculator.Calculate() の結果から取得した PhonemeTimingEntry で DurationSeconds の算出プロパティ不変条件を検証
   - 計算結果が構造の契約を満たしていることを統合的に確認（T1-1 は計算精度の検証、本テストは構造契約の検証が目的）
   - 入力: phonemeIds = [1, 10, 0, 12, 0, 10, 2], durations = [2.0f, 8.0f, 1.0f, 5.0f, 1.0f, 12.0f, 1.0f]（BOS/PAD/EOS 混在パターン）
   - puaTokenMapper: null
   - 全エントリに対して: Assert.AreEqual(entry.EndSeconds - entry.StartSeconds, entry.DurationSeconds, 1e-7f)

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

1. **PhonemeTimingEntry のコンストラクタ設計**: P1-1 の仕様で DurationSeconds は算出プロパティ（`=> EndSeconds - StartSeconds`）、コンストラクタは3引数（phoneme, startSeconds, endSeconds）と確定済み。不変条件テストは冗長に見えるが、算出プロパティの契約文書化として価値がある
2. **readonly struct の等値性**: C# の readonly struct は自動的に値ベースの Equals を生成するが、カスタム Equals/GetHashCode が実装される場合はそちらをテストする必要がある
3. **float 比較の精度**: DurationSeconds == EndSeconds - StartSeconds の検証で、浮動小数点の丸め誤差に注意。delta=1e-7f は readonly struct の内部計算では十分だが、TimingCalculator 経由の場合は累積誤差で delta を緩める必要がある可能性

### 5.2 レビューチェックリスト

- [ ] PhonemeTimingEntry の全プロパティ（Phoneme, StartSeconds, EndSeconds, DurationSeconds）がテストされているか
- [ ] DurationSeconds == EndSeconds - StartSeconds の不変条件が複数パターンで検証されているか
- [ ] default(PhonemeTimingEntry) のテストがあるか
- [ ] readonly struct であることの型レベル検証（コンパイル時保証のためテスト不要だが、コメントで明記）
- [ ] T1-1 の CreateTimingTestPhonemeIdMap() を利用した統合テスト（DurationConsistency_AfterTimingCalculation）があるか
- [ ] delta 値が適切か（直接構築テスト: 1e-7f、TimingCalculator 経由: 1e-5f）

## 6. 一から作り直すとしたら

PhonemeTimingEntry は readonly struct という最もシンプルなデータ型のため、テストも構造テスト（プロパティ値の正確性・不変条件）に絞るのが最適。P1-1 の仕様で DurationSeconds は算出プロパティ（`=> EndSeconds - StartSeconds`）と確定しているため、不変条件テスト（DurationSeconds == EndSeconds - StartSeconds）は本質的に常に成立する。それでも算出プロパティの契約を文書化し、将来のリファクタリング時の回帰検知として価値がある。

## 7. 後続タスクへの連絡事項

- 本チケットの依存先は P1-1（PhonemeTimingEntry 定義）であり、P1-4（TimingCalculator 実装）ではない。ただし DurationConsistency_AfterTimingCalculation テストメソッドは TimingCalculator を使用するため、実行時には P1-4 も必要

- piper-plus の PhonemeTimingEntry は `record` で定義されているが、uPiper では `readonly struct` を使用する（GC 圧力回避のため Unity 最適化）。テストではこの型の違いを意識する
