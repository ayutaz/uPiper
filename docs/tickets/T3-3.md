# T3-3: durations 非対応モデルテスト

## 1. タスク目的とゴール

TTSSynthesisOrchestrator の `SynthesizeWithTimingCoreAsync()` が durations 非対応モデル（InferenceOutput.HasDurations == false）の場合に Timings == null を返し、例外を投げないことを検証する。P3-2 で TTSSynthesisOrchestrator に TimingCalculator 統合が追加され、`SynthesizeWithTimingCoreAsync()` メソッドが新設される。本チケットでは durations が存在しない場合のフォールバックパスが安全に動作することを保証する。

**ゴール**: StubInferenceAudioGenerator の DurationsToReturn を default（null）に設定した状態で `SynthesizeWithTimingAsync()` を呼び出し、Timings が null であること、AudioClip が正常に生成されること、例外が発生しないことを保証する。

## 2. 実装する内容の詳細

### テストファイル
`Assets/uPiper/Tests/Editor/AudioGeneration/TTSSynthesisOrchestratorTests.cs`（既存ファイルに追加）

### 前提知識

P3-2 で TTSSynthesisOrchestrator に追加される変更:
- `SynthesizeWithTimingCoreAsync(SynthesisRequest, CancellationToken)` private メソッド新設（戻り値: `CoreSynthesisResult`）
- `SynthesizeWithTimingAsync(SynthesisRequest, CancellationToken)` internal メソッド新設（戻り値: `SynthesisWithTimingResult`）
- SynthesisWithTimingResult: `{ AudioClip, IReadOnlyList<PhonemeTimingEntry> Timings (nullable), TotalDurationSeconds, HasTimings }`
- durations 対応時: InferenceOutput.Durations を TimingCalculator.Calculate() でタイミング変換
- durations 非対応時: `Timings = null` で SynthesisWithTimingResult を構築（TimingCalculator 呼び出しをスキップ）
- 既存の `SynthesizeAsync()` は内部で SynthesizeWithTimingCoreAsync を呼び出し、AudioClip のみ返却（外部インターフェース変更なし）

P2-4 で StubInferenceAudioGenerator に追加される変更:
- `DurationsToReturn = null`（デフォルト）で InferenceOutput.HasDurations == false を返す
- `SupportsDurations = false`（デフォルト）

SynthesisWithTimingResult の構造（P1-3 で定義）:
```csharp
public sealed class SynthesisWithTimingResult
{
    public AudioClip AudioClip { get; }
    public IReadOnlyList<PhonemeTimingEntry> Timings { get; }  // nullable
    public float TotalDurationSeconds { get; }
    public bool HasTimings => Timings != null;
    internal SynthesisWithTimingResult(AudioClip audioClip, IReadOnlyList<PhonemeTimingEntry> timings, float totalDurationSeconds);
}
```

   - Arrange: StubGenerator の DurationsToReturn = float[N]（有効な durations データ）

   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: result.Timings != null

   - Assert: clip != null
   - Assert: _stubGenerator.GenerateCallCount == 1
   - 既存の SynthesizeAsync が durations 非対応でも影響を受けないこと（リグレッションテスト）

### SetUp の変更

既存の `[SetUp]` メソッドは変更不要。StubGenerator のデフォルト状態（DurationsToReturn = null）が durations 非対応モデルのシミュレーションとして十分。各テストメソッド内で必要に応じて DurationsToReturn を設定する。

### コードパターン例

```csharp
[Test]
public async Task SynthesizeWithTimingAsync_NoDurations_TimingsIsNull()
{
    // Arrange: DurationsToReturn = null (デフォルト) → durations 非対応モデル
    var config = CreateValidatedConfig(enableSilence: false);
    var orchestrator = new TTSSynthesisOrchestrator(
        _stubGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
        config, _voiceConfig);
    var request = new SynthesisRequest(
        new[] { "k", "o", "N" },
        null,
        1.0f, 0.667f, 0.8f,
        0, 0);

    // Act
    var result = await orchestrator.SynthesizeWithTimingAsync(request);

    // Assert
    Assert.IsNotNull(result, "SynthesisWithTimingResult が返ること");
    Assert.IsNull(result.Timings,
        "durations 非対応モデルでは Timings が null であること");
    Assert.IsFalse(result.HasTimings,
        "HasTimings が false であること");
    Assert.IsNotNull(result.AudioClip,
        "AudioClip は正常に生成されること");
}
```

### テスト #6 の DurationsToReturn 設定

durations 対応モデルの対比テスト（#6）では、DurationsToReturn の長さが PhonemeEncoder のエンコード結果の phonemeIds.Length と一致する必要がある。PhonemeEncoder は BOS + PAD intersperse + EOS を付加するため:

```csharp
// phonemes = ["k", "o", "N"] → phonemeIds: [BOS=1, PAD=0, k=8, PAD=0, o=7, PAD=0, N=16, PAD=0, EOS=2] = 9要素
// intersperse PAD あり（PhonemeType == null かつ VoiceId に "ja_JP" を含まない）
// → BOS + PAD + 3*(phoneme + PAD) + EOS = 1 + 1 + 3*2 + 1 = 9
_stubGenerator.DurationsToReturn = new float[9];
for (var i = 0; i < 9; i++)
    _stubGenerator.DurationsToReturn[i] = (float)(i + 1);

// テストでは SynthesizeWithTimingAsync を使用（SynthesizeWithTimingCoreAsync は private）
var result = await orchestrator.SynthesizeWithTimingAsync(request);
```

## 6. 一から作り直すとしたら

テスト設計は「durations 非対応モデルのフォールバックパスが安全か」に絞るのが最適。TTSSynthesisOrchestrator の durations 対応パス（TimingCalculator 統合）は T4-1/T4-2 の統合テストで十分にカバーされるため、本チケットでは非対応パスのみに集中する。

テスト #6（durations 対応の対比テスト）は本来 T4-1 のスコープだが、「非対応時に Timings == null」の検証に「対応時に Timings != null」の対比を含めることで、条件分岐の両方のブランチが本チケット内で確認できる。これにより P3-2 のレビュー時に非対応パスの安全性を独立して検証可能になる。

Prosody 付き・句分割付きの非対応テスト（#4, #5）は、TTSSynthesisOrchestrator 内の3分岐（通常/Prosody/句分割）全てで durations 非対応が安全に動作することを保証するために必要。

注意: テストでは `SynthesizeWithTimingAsync`（internal メソッド）を使用する。`SynthesizeWithTimingCoreAsync` は private readonly struct `CoreSynthesisResult` を返す private メソッドのため、テストから直接呼び出さない。

## 7. 後続タスクへの連絡事項

- P3-2 実装時: `SynthesizeWithTimingCoreAsync()` で `InferenceOutput.HasDurations == false` の場合、TimingCalculator.Calculate() を呼び出さずに `Timings = null` で CoreSynthesisResult を構築すること。null チェックなしで TimingCalculator に渡すと NullReferenceException が発生する
- P3-2 実装時: `SynthesizeWithTimingCoreAsync()` のシグネチャは `(SynthesisRequest request, CancellationToken cancellationToken = default)` であり、`string[] phonemes` 引数は不要（TimingCalculator は phonemeIds + phonemeIdMap の逆引きで音素文字列を取得する）
- P3-2 実装時: 句分割パス（SplitInferenceOrchestrator 経由）でも durations 非対応を正しくハンドルすること。SplitInferenceOrchestrator の InferenceOutput.HasDurations == false の場合、結合後の InferenceOutput も HasDurations == false とする（P3-1/T3-1 と連携）
- T4-4（統合テスト: durations 非対応）: PiperTTS.SynthesizeWithTimingAsync 経由での durations 非対応テスト。本チケットは TTSSynthesisOrchestrator 単体での非対応テスト
- T4-5（リグレッションテスト）: 既存の SynthesizeAsync が durations 対応後も影響を受けないことの網羅的検証。本チケットのテスト #7 はその先行検証
