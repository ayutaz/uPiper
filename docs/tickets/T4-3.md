# T4-3: 統合テスト -- キャッシュ整合

## 1. タスク目的とゴール

SynthesizeWithTimingAsync() の 1 回目（キャッシュミス）と 2 回目（キャッシュヒット）でタイミング情報が同一であることを検証する。P3-3 で AudioSynthesisCache が PhonemeTimingEntry[] をオプショナルに格納するよう拡張され、P3-2 で TTSSynthesisOrchestrator がキャッシュにタイミング情報を格納するようになる。本チケットではキャッシュ経由のタイミング復元の正確性、および推論の呼び出し回数（2 回目はスキップ）を保証する。

**ゴール**: 同一 SynthesisRequest に対する 2 回の SynthesizeWithTimingAsync() 呼び出しで、1 回目はキャッシュミス（GenerateCallCount 増加）、2 回目はキャッシュヒット（GenerateCallCount 不変）となり、両者の Timings が要素数・各エントリの Phoneme/StartSeconds/EndSeconds まで完全に一致することを保証する。

## 2. 実装する内容の詳細

### テストファイル
`Assets/uPiper/Tests/Editor/PiperTTSSynthesizeWithTimingTests.cs`（T4-1 で作成するファイルに追加）

### 前提知識

P3-3 で AudioSynthesisCache に追加される変更:
- `Set(key, samples, sampleRate, timings)`: PhonemeTimingEntry[] を格納
- `TryGet(key, out samples, out sampleRate, out timings)`: PhonemeTimingEntry[] を復元

P3-2 で TTSSynthesisOrchestrator に追加される変更:
- SynthesizeWithTimingAsync() 内でキャッシュヒット時に格納済みの timings を復元
- キャッシュミス時に推論 + TimingCalculator 計算 + キャッシュ格納

キャッシュキー: AudioSynthesisCache.GenerateKey(phonemeIds, prosodyFlat, lengthScale, noiseScale, noiseW, speakerId, languageId)

### テスト戦略

TTSSynthesisOrchestrator のコンストラクタに AudioSynthesisCache インスタンスを渡し、キャッシュ付きの統合テストを実施する。1 回目の呼び出しでキャッシュに格納、2 回目でキャッシュヒットを検証する。

### 実装するテストメソッド

1. **SynthesizeWithTimingAsync_SecondCall_CacheHit_GenerateNotCalled**
   - Arrange:
     - AudioSynthesisCache インスタンスを生成
     - StubGenerator.DurationsToReturn = float[9]（有効な durations）
     - TTSSynthesisOrchestrator を audioCache 付きで構築
     - SynthesisRequest: phonemes = ["k", "o", "N"]
   - Act:
     - 1 回目: `orchestrator.SynthesizeWithTimingAsync(request)`
     - Assert: _stubGenerator.GenerateCallCount == 1（キャッシュミス）
     - 2 回目: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: _stubGenerator.GenerateCallCount == 1（2 回目はキャッシュヒットで推論スキップ）

2. **SynthesizeWithTimingAsync_SecondCall_CacheHit_TimingsCountEqual**
   - Arrange: 同上（キャッシュ付き）
   - Act:
     - result1 = 1 回目呼び出し
     - result2 = 2 回目呼び出し
   - Assert: result1.Timings.Count == result2.Timings.Count

3. **SynthesizeWithTimingAsync_SecondCall_CacheHit_TimingsValuesIdentical**
   - Arrange: 同上（キャッシュ付き）
   - Act:
     - result1 = 1 回目呼び出し
     - result2 = 2 回目呼び出し
   - Assert: 全エントリ i について:
     - result1.Timings[i].Phoneme == result2.Timings[i].Phoneme
     - result1.Timings[i].StartSeconds == result2.Timings[i].StartSeconds (delta=1e-5f)
     - result1.Timings[i].EndSeconds == result2.Timings[i].EndSeconds (delta=1e-5f)

4. **SynthesizeWithTimingAsync_SecondCall_CacheHit_HasTimingsTrue**
   - Arrange: 同上（キャッシュ付き）
   - Act:
     - 1 回目呼び出し（キャッシュ格納）
     - result2 = 2 回目呼び出し（キャッシュヒット）
   - Assert: result2.HasTimings == true
   - Assert: result2.AudioClip != null
   - キャッシュヒットでもタイミング情報と AudioClip が正常に返ること

5. **SynthesizeWithTimingAsync_DifferentRequests_IndependentCacheEntries**
   - Arrange:
     - AudioSynthesisCache インスタンスを生成
     - StubGenerator.DurationsToReturn = float[9]
     - request1: phonemes = ["k", "o", "N"], lengthScale = 1.0f
     - request2: phonemes = ["k", "o", "N"], lengthScale = 2.0f（異なるパラメータ）
   - Act:
     - result1 = orchestrator.SynthesizeWithTimingAsync(request1)
     - result2 = orchestrator.SynthesizeWithTimingAsync(request2)
   - Assert: _stubGenerator.GenerateCallCount == 2（異なるキャッシュキー → 両方ミス）
   - Assert: result1.HasTimings == true && result2.HasTimings == true

6. **SynthesizeWithTimingAsync_CacheCleared_SecondCallMiss**
   - Arrange:
     - AudioSynthesisCache インスタンスを生成
     - StubGenerator.DurationsToReturn = float[9]
     - 1 回目: `orchestrator.SynthesizeWithTimingAsync(request)`（キャッシュ格納）
   - Act:
     - cache.Clear()
     - 2 回目: `orchestrator.SynthesizeWithTimingAsync(request)`（キャッシュクリア後）
   - Assert: _stubGenerator.GenerateCallCount == 2（Clear 後は再推論）
   - Assert: result2.HasTimings == true

### コードパターン例

```csharp
[Test]
public async Task SynthesizeWithTimingAsync_SecondCall_CacheHit_TimingsValuesIdentical()
{
    // Arrange
    _stubGenerator.DurationsToReturn = new[]
    {
        0.01f, 0.02f, 0.03f, 0.02f, 0.05f, 0.02f, 0.04f, 0.02f, 0.01f
    };
    var cache = new AudioSynthesisCache();
    var config = CreateValidatedConfig(enableSilence: false);
    var orchestrator = new TTSSynthesisOrchestrator(
        _stubGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
        config, _voiceConfig, audioCache: cache, puaTokenMapper: _puaTokenMapper);
    var request = new SynthesisRequest(
        new[] { "k", "o", "N" },
        null,
        1.0f, 0.667f, 0.8f,
        0, 0);

    // Act
    var result1 = await orchestrator.SynthesizeWithTimingAsync(request);
    var result2 = await orchestrator.SynthesizeWithTimingAsync(request);

    // Assert
    Assert.AreEqual(1, _stubGenerator.GenerateCallCount,
        "2 回目はキャッシュヒットで推論がスキップされること");
    Assert.AreEqual(result1.Timings.Count, result2.Timings.Count,
        "タイミングエントリ数が一致すること");

    for (var i = 0; i < result1.Timings.Count; i++)
    {
        Assert.AreEqual(result1.Timings[i].Phoneme, result2.Timings[i].Phoneme,
            $"Timings[{i}].Phoneme が一致すること");
        Assert.AreEqual(result1.Timings[i].StartSeconds, result2.Timings[i].StartSeconds, 1e-5f,
            $"Timings[{i}].StartSeconds が一致すること");
        Assert.AreEqual(result1.Timings[i].EndSeconds, result2.Timings[i].EndSeconds, 1e-5f,
            $"Timings[{i}].EndSeconds が一致すること");
    }
}
```

## 6. 一から作り直すとしたら

テスト設計は「キャッシュミス/ヒットの動作保証」と「Timings の値レベルの一致」の 2 軸に集中するのが最適。AudioSynthesisCache 単体のテスト（T3-2）で格納/復元の基本動作は保証されているため、本チケットでは TTSSynthesisOrchestrator を介した統合動作に特化する。

テスト #3（TimingsValuesIdentical）が最も重要。キャッシュから復元された Timings が元の計算結果と完全に一致することは、ユーザーが 2 回目の呼び出しで異なるタイミング情報を受け取る事態を防ぐ。PhonemeTimingEntry が readonly struct であるため値の不変性は型レベルで保証されているが、配列参照の共有/コピーの違いによる潜在的な問題をテストで検出する。

テスト #5（異なるリクエストの独立性）は、キャッシュキーの衝突が発生しないことの間接的な検証。AudioSynthesisCache.GenerateKey の FNV-1a ハッシュが異なるパラメータに対して異なるキーを生成することを前提とする。

## 7. 後続タスクへの連絡事項

- P3-2 実装時: TTSSynthesisOrchestrator の SynthesizeWithTimingAsync() でキャッシュヒット時に `_audioCache.TryGet(key, out samples, out sampleRate, out timings)` の timings を SynthesisWithTimingResult に直接格納すること。TimingCalculator の再計算は不要
- P3-2 実装時: キャッシュ格納時に `_audioCache.Set(key, audioData, sampleRate, timings?.ToArray())` の形式で呼び出すこと。timings が null の場合（durations 非対応モデル）は null がそのまま格納される
- P3-3 実装時: TryGet で返す PhonemeTimingEntry[] が Set 時と同一の参照であることを文書化すること。TTSSynthesisOrchestrator が防御的コピーを行うかどうかは P3-2 の設計に委ねる
- T4-5（リグレッション）: PiperTTS.ClearCache() がタイミング情報も含めてクリアすることの検証は T4-5 で対応
