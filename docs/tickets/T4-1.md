# T4-1: 統合テスト -- タイミング付き合成

## 1. タスク目的とゴール

PiperTTS.SynthesizeWithTimingAsync() がエンドツーエンドでタイミング情報付きの音声合成結果を返すことを検証する。P4-2 で PiperTTS.Inference.cs に SynthesizeWithTimingAsync() が実装され、IPiperTTS インターフェース経由で呼び出し可能になる。本チケットでは StubInferenceAudioGenerator の DurationsToReturn を設定して既知の durations データを注入し、SynthesisWithTimingResult の Timings エントリ数が 0 より大きいこと、各エントリの StartSeconds/EndSeconds の値が妥当であることを保証する。

**ゴール**: SynthesizeWithTimingAsync() が正常に SynthesisWithTimingResult を返し、Timings に実音素（BOS/EOS/PAD を除く）のタイミングエントリが含まれ、StartSeconds/EndSeconds が非負かつ単調増加であることを保証する。

### 前提知識

P4-2 で PiperTTS に追加される変更:
- `SynthesizeWithTimingAsync(SynthesisRequest, CancellationToken)` public メソッド新設
- 戻り値: `Task<SynthesisWithTimingResult>`
- 内部で TTSSynthesisOrchestrator.SynthesizeWithTimingAsync() を呼び出す
- ガード句: disposed チェック、initialized チェック（既存 SynthesizeAsync と同一パターン。phonemes null/empty チェックは TTSSynthesisOrchestrator 側で実施）

SynthesisWithTimingResult の構造（P1-3 で定義）:
```csharp
public sealed class SynthesisWithTimingResult
{
    public AudioClip AudioClip { get; }
    public IReadOnlyList<PhonemeTimingEntry> Timings { get; }  // nullable
    public float TotalDurationSeconds { get; }
    public bool HasTimings => Timings != null;
}
```

PhonemeTimingEntry の構造（P1-1 で定義）:
```csharp
public readonly struct PhonemeTimingEntry
{
    public string Phoneme { get; }
    public float StartSeconds { get; }
    public float EndSeconds { get; }
    public float DurationSeconds => EndSeconds - StartSeconds;
}
```

P2-4 で StubInferenceAudioGenerator に追加される変更:
- `DurationsToReturn` プロパティ（`float[]` 型）で固定 durations データを注入可能
- `GenerateAudioAsync` の戻り値が `Task<InferenceOutput>` に変更

### テスト戦略

PiperTTSSynthesizeTests.cs と同様に PiperTTS インスタンスを直接構築するが、InitializeWithInferenceAsync() は内部で実 ONNX モデルを必要とするため、テストでは TTSSynthesisOrchestrator レベルで StubGenerator を注入する。テストクラスではリフレクションまたは internal フィールドアクセス（InternalsVisibleTo）を使用して _orchestrator を差し替えるか、TTSSynthesisOrchestrator を直接テストする。

既存の TTSSynthesisOrchestratorTests.cs のパターンに準じ、TTSSynthesisOrchestrator.SynthesizeWithTimingAsync() を直接呼び出すアプローチを採用する。PiperTTS レベルのガード句テストは T4-5 で実施する。

### 実装するテストメソッド

1. **SynthesizeWithTimingAsync_WithDurations_TimingsNotEmpty**
   - Arrange: StubGenerator.DurationsToReturn = float[9]（phonemes ["k", "o", "N"] のエンコード後 9 要素分）
   - 各要素に正の値を設定（例: [0.01f, 0.02f, 0.03f, 0.02f, 0.05f, 0.02f, 0.04f, 0.02f, 0.01f]）
   - config: enableSilence = false
   - SynthesisRequest: phonemes = ["k", "o", "N"], prosodyFlat = null
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: result.HasTimings == true
   - Assert: result.Timings.Count > 0
   - Assert: result.Timings.Count == 3（BOS/EOS/PAD を除いた実音素数）

2. **SynthesizeWithTimingAsync_WithDurations_StartSecondsNonNegative**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: 全エントリの StartSeconds >= 0.0f

3. **SynthesizeWithTimingAsync_WithDurations_EndSecondsGreaterThanStart**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: 全エントリの EndSeconds > StartSeconds（DurationSeconds > 0）

4. **SynthesizeWithTimingAsync_WithDurations_TimingsMonotonicallyIncreasing**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: Timings[i].StartSeconds >= Timings[i-1].EndSeconds（i >= 1 の全エントリ）
   - タイミングが時系列順に並んでいることを検証

5. **SynthesizeWithTimingAsync_WithDurations_PhonemeNamesPreserved**
   - Arrange: 同上（phonemes = ["k", "o", "N"]）
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: Timings の Phoneme フィールドが入力音素の順序と一致
   - TimingCalculator が phonemeIdMap の逆引きで正しい音素名を復元していること

6. **SynthesizeWithTimingAsync_WithDurations_AudioClipNotNull**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: result.AudioClip != null
   - Assert: result.TotalDurationSeconds > 0

7. **SynthesizeWithTimingAsync_WithDurations_FirstEntryStartsAtZero**
   - Arrange: 同上
   - Act: `orchestrator.SynthesizeWithTimingAsync(request)`
   - Assert: result.Timings[0].StartSeconds == 0.0f (delta=1e-5f)
   - 最初の音素のタイミングが 0 秒から開始すること

### コードパターン例

```csharp
[Test]
public async Task SynthesizeWithTimingAsync_WithDurations_TimingsNotEmpty()
{
    // Arrange
    _stubGenerator.DurationsToReturn = new[]
    {
        0.01f, 0.02f, 0.03f, 0.02f, 0.05f, 0.02f, 0.04f, 0.02f, 0.01f
    };
    var config = CreateValidatedConfig(enableSilence: false);
    var orchestrator = new TTSSynthesisOrchestrator(
        _stubGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
        config, _voiceConfig, audioCache: null, puaTokenMapper: _puaTokenMapper);
    var request = new SynthesisRequest(
        new[] { "k", "o", "N" },
        null,
        1.0f, 0.667f, 0.8f,
        0, 0);

    // Act
    var result = await orchestrator.SynthesizeWithTimingAsync(request);

    // Assert
    Assert.IsTrue(result.HasTimings, "タイミング情報が存在すること");
    Assert.Greater(result.Timings.Count, 0,
        "Timings エントリ数が 0 より大きいこと");
    Assert.AreEqual(3, result.Timings.Count,
        "BOS/EOS/PAD を除いた実音素数（k, o, N）が 3 であること");
}
```

### DurationsToReturn の長さ計算

PhonemeEncoder は phonemes ["k", "o", "N"] を以下のようにエンコードする（PhonemeType == null, intersperse PAD あり）:
```
[BOS=1, PAD=0, k=8, PAD=0, o=7, PAD=0, N=16, PAD=0, EOS=2] = 9要素
```
DurationsToReturn は phonemeIds.Length と同じ 9 要素に設定する。TimingCalculator は BOS/EOS/PAD の duration をスキップし、実音素（k, o, N）の duration のみからタイミングを計算する。

## 6. 一から作り直すとしたら

テスト設計は「SynthesizeWithTimingAsync の基本契約（Timings が非空・妥当な値・単調増加）」に絞るのが最適。タイミングの数値精度（T4-2）、キャッシュ整合（T4-3）、durations 非対応（T4-4）は別チケットで個別に検証するため、本チケットでは「タイミング情報が正しい構造で返されること」の基本保証に集中する。

TTSSynthesisOrchestrator レベルでテストするアプローチは、PiperTTS の InitializeWithInferenceAsync() が実 ONNX モデルを必要とする制約を回避しつつ、推論パイプライン全体（PhonemeEncoder -> Generator -> TimingCalculator）の統合動作を検証できる。PiperTTS レベルのテスト（ガード句、Dispose 後の呼び出し等）は T4-5 で対応する。

DurationsToReturn に設定する値は任意だが、BOS/EOS/PAD 位置（インデックス 0, 1, 3, 5, 7, 8）の値は TimingCalculator でスキップされるため、実音素位置（インデックス 2, 4, 6）の値が Timings エントリに反映される。テストではこの対応関係を意識した値設定を行う。
