# P1-2: InferenceOutput 定義

## 1. タスク目的とゴール

ONNX推論の生出力（音声波形 + 音素フレーム持続時間）をまとめて保持する内部構造体を定義する。現在の `InferenceAudioGenerator.ExtractResults()` は `NativeArray<float>`（音声データのみ）を返すが、フォネムタイミング機能のためには `durations` テンソル（各音素のスペクトログラムフレーム数）も同時に取得する必要がある。`InferenceOutput` はこの2つの `NativeArray<float>` をまとめ、`IDisposable` パターンで安全に破棄する。

### なぜ必要か

- Phase 2（P2-1）で `IInferenceAudioGenerator.GenerateAudioAsync` の戻り値型を `NativeArray<float>` から `InferenceOutput` に変更する。その前に型定義が必要
- `NativeArray` のライフサイクル管理（Allocator.Persistent で確保、using で破棄）を1か所に集約し、リーク防止を構造的に保証する
- durations 非対応モデル（`outputs.Count == 1`）では `Durations.IsCreated == false` で判定する設計。この分岐ロジックを型レベルで表現する

### 2.2 型定義

```csharp
using System;
using Unity.Collections;

namespace uPiper.Core.AudioGeneration
{
    /// <summary>
    /// ONNX推論の生出力を保持する内部構造体。
    /// <c>Audio</c>（音声波形）と <c>Durations</c>（音素フレーム持続時間）の
    /// 2つの <see cref="NativeArray{T}"/> をまとめて管理し、
    /// <see cref="IDisposable"/> パターンで安全に破棄する。
    /// </summary>
    /// <remarks>
    /// <para>
    /// <c>Durations</c> はモデルが <c>durations</c> 出力テンソルを持つ場合のみ有効。
    /// 非対応モデルでは <c>Durations.IsCreated == false</c> となる。
    /// </para>
    /// <para>
    /// 両 <see cref="NativeArray{T}"/> は <c>Allocator.Persistent</c> で確保されることを想定。
    /// 呼び出し元は <c>using</c> ステートメントまたは明示的な <see cref="Dispose"/> で
    /// リソースを解放すること。
    /// </para>
    /// </remarks>
    internal readonly struct InferenceOutput : IDisposable
    {
        /// <summary>
        /// 音声波形データ（float32, モノラル）。
        /// <see cref="InferenceAudioGenerator.ExtractResults"/> で確保された
        /// <c>Allocator.Persistent</c> の <see cref="NativeArray{T}"/>。
        /// </summary>
        public NativeArray<float> Audio { get; }

        /// <summary>
        /// 音素ごとのフレーム持続時間。
        /// モデルの <c>durations</c> 出力テンソル（shape: <c>[1, phoneme_length]</c>）から読み取った値。
        /// <c>durations[i]</c> は <c>phonemeIds[i]</c> に対応するスペクトログラムフレーム数。
        /// </summary>
        /// <remarks>
        /// モデルが <c>durations</c> 出力を持たない場合は <c>IsCreated == false</c>。
        /// <see cref="HasDurations"/> で判定可能。
        /// </remarks>
        public NativeArray<float> Durations { get; }

        /// <summary>
        /// durations データが利用可能かどうか。
        /// </summary>
        public bool HasDurations => Durations.IsCreated;

        /// <summary>
        /// InferenceOutput を構築する。
        /// </summary>
        /// <param name="audio">
        /// 音声波形データ。<c>Allocator.Persistent</c> で確保された <see cref="NativeArray{T}"/>。
        /// </param>
        /// <param name="durations">
        /// 音素フレーム持続時間。モデル非対応時は <c>default</c> を渡す。
        /// </param>
        internal InferenceOutput(NativeArray<float> audio, NativeArray<float> durations)
        {
            Audio = audio;
            Durations = durations;
        }

        /// <summary>
        /// <see cref="Audio"/> と <see cref="Durations"/> の両 <see cref="NativeArray{T}"/> を
        /// 安全に破棄する。未作成（<c>IsCreated == false</c>）の配列は無視する。
        /// </summary>
        public void Dispose()
        {
            if (Audio.IsCreated)
                Audio.Dispose();
            if (Durations.IsCreated)
                Durations.Dispose();
        }
    }
}
```

### 2.3 設計判断の根拠

| 判断 | 理由 |
|------|------|
| `internal` | 推論レイヤーの内部型。公開APIには `SynthesisWithTimingResult`（P1-3）を通じて露出する。外部パッケージから参照されることはない |
| `readonly struct` | `NativeArray<float>` は構造体であり、`InferenceOutput` も値型にすることで不要なヒープアロケーションを回避。`SynthesisRequest` パターンと整合 |
| `IDisposable` | `NativeArray<float>` は `Allocator.Persistent` で確保されるため、明示的な Dispose が必須。`using` ステートメントでの利用を想定 |
| `NativeArray<float>` (Durations) | Sentis API の `DownloadToNativeArray()` が `NativeArray<float>` を返すため、audio と同じパターンで読み取り可能。`float[]` へのコピーは TimingCalculator（P1-4）側で実施 |
| `HasDurations` プロパティ | `Durations.IsCreated` の意図を明示化。`PhonemizeResult.HasProsody` / `SynthesisRequest.HasProsody` パターンと整合 |
| `internal` コンストラクタ | `InferenceAudioGenerator.ExtractResults()` のみが構築する。テストアセンブリからは `InternalsVisibleTo` でアクセス可能 |
| durations 未対応時は `default` NativeArray | `NativeArray<float>?` (nullable) は値型のboxingを伴う。`IsCreated == false` で判定するパターンは `AudioNormalizer.NormalizeInPlace` (L54) 等で既に使用されている |

### 2.5 piper-plus との対応

| piper-plus (`SynthesisResult`) | uPiper (`InferenceOutput`) | 差分理由 |
|---|---|---|
| `record SynthesisResult(short[] Audio, float[]? Durations)` (参照型, managed配列) | `internal readonly struct InferenceOutput` (値型, NativeArray) | Unity/Sentis では NativeArray がネイティブ。managed 配列コピーを回避 |
| `public` | `internal` | 推論レイヤー内部型。公開APIは `SynthesisWithTimingResult` |
| `float[]? Durations` (nullable) | `NativeArray<float> Durations` + `HasDurations` | NativeArray は構造体のため nullable 不可。`IsCreated` で判定 |
| IDisposable なし | `IDisposable` 実装 | NativeArray は明示的 Dispose が必須 |

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

| # | 懸念 | 対策 |
|---|------|------|
| 1 | `readonly struct` + `IDisposable` の defensive copy 問題。`readonly` フィールドに格納された `InferenceOutput` に対して `Dispose()` を呼ぶと defensive copy が作られ、元のNativeArrayが破棄されない | 使用箇所では `using var output = ...` パターンを徹底する。`readonly` フィールドに格納しない。既存の `InferenceContext`（InferenceAudioGenerator.cs L501-539）も同様に `using var ctx = PrepareInputs(...)` で使用されており、同じパターンを踏襲 |
| 2 | NativeArray の二重 Dispose | `NativeArray.Dispose()` は二重呼び出し時に例外をスローする。`InferenceOutput.Dispose()` は `IsCreated` チェック後に Dispose するため安全だが、外部から `output.Audio.Dispose()` を先に呼ばれると `InferenceOutput.Dispose()` で例外が発生する。ドキュメントで「Dispose は InferenceOutput 経由で行うこと」を明記 |
| 3 | `Allocator.Persistent` の NativeArray が Dispose されずリーク | `using` 必須をドキュメントとコードコメントで明示。Phase 2 の `ExecuteInference()` で `try/catch` 内に Dispose を配置（既存の audio Dispose パターン L372-374 と同様） |
| 4 | `internal` のため外部パッケージからテストできない | テストアセンブリは `InternalsVisibleTo` 属性で `internal` にアクセス可能。`uPiper.Runtime.asmdef` の `internalsVisibleTo` に `uPiper.Tests.Editor` / `uPiper.Tests.Runtime` が含まれていることを確認済み |

## 6. 一から作り直すとしたら

### 代替案A: class + IDisposable（参照型）

```csharp
internal sealed class InferenceOutput : IDisposable
{
    private bool _disposed;
    public NativeArray<float> Audio { get; }
    public NativeArray<float> Durations { get; }

    public void Dispose()
    {
        if (_disposed) return;
        _disposed = true;
        if (Audio.IsCreated) Audio.Dispose();
        if (Durations.IsCreated) Durations.Dispose();
    }
}
```

**メリット**: defensive copy 問題がない。`_disposed` フラグで二重 Dispose を完全に防止。`InferenceContext`（L501-539）と同じ sealed class パターン。
**デメリット**: ヒープアロケーション1回（GC圧力）。推論のたびに `new InferenceOutput(...)` が GC 対象になる。

**トレードオフ分析**: `InferenceOutput` の生存期間は短い（`ExecuteInference` メソッド内の数行）。GC 圧力は Gen0 で即回収されるため実害は軽微。一方、defensive copy リスクは `using var` パターンで回避可能であり、既存コードベースでも `using var ctx = PrepareInputs(...)` で同パターンを使用済み。

**判定**: 要件書（requirements.md L69-82）で `internal readonly struct` と明記されており、既存設計方針に従う。defensive copy 問題は使用パターンの規約で回避する。

### 代替案B: Tuple 戻り値

```csharp
// ExtractResults の戻り値を Tuple にする
private (NativeArray<float> Audio, NativeArray<float> Durations) ExtractResults()
```

**メリット**: 新しい型定義が不要。コード量最小。
**デメリット**: IDisposable パターンを適用できない。呼び出し元が2つの NativeArray を個別に Dispose する責務を負い、片方の Dispose 漏れリスクが高まる。型名がないためコードの可読性が低下。

**判定**: リソース管理の安全性を優先し、専用型を定義する。

### 代替案C: Durations を `float[]` (managed配列) にする

```csharp
internal readonly struct InferenceOutput : IDisposable
{
    public NativeArray<float> Audio { get; }
    public float[] Durations { get; }  // managed 配列
    // ...
}
```

**メリット**: Durations の Dispose が不要（GCが回収）。nullable `float[]?` で durations 非対応を表現可能。
**デメリット**: `NativeArray<float>.Copy` → `float[]` のコピーが必要。Audio との対称性が失われる。将来的に durations を NativeArray のまま下流に渡す最適化ができなくなる。

**判定**: Audio と Durations の対称性を維持し、将来の最適化余地を確保するため、両方とも `NativeArray<float>` にする。managed 配列への変換は `TimingCalculator`（P1-4）が必要に応じて実施する。

## 7. 後続タスクへの連絡事項

### P2-1 (IInferenceAudioGenerator 戻り値型変更) への連絡

- `IInferenceAudioGenerator.GenerateAudioAsync` の戻り値を `Task<NativeArray<float>>` から `Task<InferenceOutput>` に変更する
- `InferenceOutput` は `internal` のため、`IInferenceAudioGenerator` も `internal` のままであれば問題ない。ただし現在は `public interface` であるため、戻り値型を `internal` 型にするとコンパイルエラーになる。対策:
  - 案1: `IInferenceAudioGenerator` を `internal` に変更（推論レイヤー内部で完結するため公開不要）
  - 案2: `InferenceOutput` を `public` に変更
  - **推奨**: 案1。`IInferenceAudioGenerator` の実装クラス（`InferenceAudioGenerator`, `ShortTextMitigatingGenerator`）は全て `internal` 使用。公開 API は `IPiperTTS` 経由。ただしこの判断は P2-1 担当者が既存コードの参照範囲を確認して最終決定すること

### P2-2 (InferenceAudioGenerator durations 読み取り) への連絡

- `ExtractResults()` の戻り値を `InferenceOutput` に変更
- Durations の読み取りパターンは Audio と同一: `PeekOutput` → `ReadbackAndClone` → `DownloadToNativeArray` → `NativeArray<float>.Copy`
- durations 非対応モデル: `Durations` に `default(NativeArray<float>)` を渡す
- `ExecuteInference()` (L337-378) の既存 NativeArray Dispose パターン（L372-374 の `if (audioData.IsCreated) audioData.Dispose()`）を `InferenceOutput.Dispose()` に置き換え

### P2-3 (ShortTextMitigatingGenerator 追従) への連絡

- `ShortTextMitigatingGenerator` が `IInferenceAudioGenerator` をデコレートしているため、戻り値型の変更に追従が必要
- パディング除去ロジック: `InferenceOutput.Durations` から PAD 位置の要素を除去した新しい `NativeArray<float>` を構築し、元の `InferenceOutput` を Dispose して新しい `InferenceOutput` を返す

### T2-2 (InferenceOutput Dispose テスト) への連絡

- テスト内で `NativeArray<float>` を `Allocator.Temp` で確保し、`InferenceOutput` 経由で Dispose されることを検証
- `default(InferenceOutput).Dispose()` が例外を投げないことを検証
- テストアセンブリは `InternalsVisibleTo` で `internal` コンストラクタにアクセス可能
- NativeArray のテストでは `try/finally` パターンを使用し、テスト失敗時もリーク防止すること（既存テストパターンに従う）

## 8. Phase 1 レビュー結果からの連絡事項

**[Phase 1 レビュー指摘]** `readonly struct` + `IDisposable` の defensive copy リスクについてレビューで指摘あり。`readonly` フィールドに格納された場合、`Dispose()` 呼び出しで defensive copy が作られ元の NativeArray が破棄されない問題。Phase 1 では型定義のみで実際の使用箇所がないため実害なし。**P2-1 実装開始時に `internal sealed class` への変換を推奨**。詳細は本チケット §6 代替案A を参照。

- 変更内容: `internal readonly struct InferenceOutput : IDisposable` → `internal sealed class InferenceOutput : IDisposable`
- `_disposed` フラグによる二重 Dispose 防止を追加
- GC 圧力は Gen0 即回収のため実害は軽微（InferenceOutput の生存期間は推論メソッド内の数行）
