# P3-2: TTSSynthesisOrchestrator タイミング計算統合

## 1. タスク目的とゴール

`TTSSynthesisOrchestrator` にタイミング計算ロジックを統合し、`InferenceOutput` の durations から `TimingCalculator` を呼び出して `SynthesisWithTimingResult` を構築する。共通コアとなる private メソッド `SynthesizeWithTimingCoreAsync()` と、外部公開用の internal メソッド `SynthesizeWithTimingAsync()` を追加する。既存の `SynthesizeAsync()` は後方互換を維持しつつ、内部で共通コアを共有する。

### なぜ必要か

- Phase 1 で `TimingCalculator`（P1-4）と `SynthesisWithTimingResult`（P1-3）が定義されたが、推論パイプラインから呼び出すオーケストレーションが未実装
- Phase 2/3 前半で `IInferenceAudioGenerator` と `ISplitInferenceOrchestrator` の戻り値が `InferenceOutput` に変更され、durations がパイプライン全体を流通するようになった。TTSSynthesisOrchestrator でこれを受け取り、タイミング情報に変換する最終統合が必要
- Phase 4（P4-1/P4-2）で `IPiperTTS.SynthesizeWithTimingAsync()` を公開するためには、TTSSynthesisOrchestrator 内にタイミング付き合成の実装が必要

### 2.2 コンストラクタ変更

`TimingCalculator` の呼び出しに必要な `PuaTokenMapper` インスタンスをコンストラクタで受け取る。

```csharp
// 追加フィールド
private readonly PuaTokenMapper _puaTokenMapper;

// コンストラクタ変更
public TTSSynthesisOrchestrator(
    IInferenceAudioGenerator generator,
    ISplitInferenceOrchestrator splitOrchestrator,
    PhonemeEncoder phonemeEncoder,
    AudioClipBuilder audioClipBuilder,
    IPiperConfigReadOnly config,
    PiperVoiceConfig voiceConfig,
    AudioSynthesisCache audioCache = null,
    PuaTokenMapper puaTokenMapper = null)  // 新規追加
{
    _generator = generator ?? throw new ArgumentNullException(nameof(generator));
    _splitOrchestrator = splitOrchestrator ?? throw new ArgumentNullException(nameof(splitOrchestrator));
    _phonemeEncoder = phonemeEncoder ?? throw new ArgumentNullException(nameof(phonemeEncoder));
    _audioClipBuilder = audioClipBuilder ?? throw new ArgumentNullException(nameof(audioClipBuilder));
    _config = config;
    _voiceConfig = voiceConfig;
    _audioCache = audioCache;
    _puaTokenMapper = puaTokenMapper; // nullable: null の場合は PUA 逆引きスキップ
}
```

**注意**: `puaTokenMapper` は nullable オプションパラメータとして追加する。既存のコンストラクタ呼び出し箇所（`PiperTTS.Inference.cs`）は P4-2 で `puaTokenMapper` を渡すように変更する。P3-2 時点では既存呼び出し箇所は変更不要（デフォルト null が適用される）。

### 2.3 内部2段構成

内部を共通コアロジック（推論 + タイミング計算）とラッパーメソッドに分割する。

#### 内部結果型（共通コアの戻り値）

```csharp
/// <summary>
/// SynthesizeWithTimingCoreAsync の内部結果。
/// AudioClip とオプションのタイミング情報を保持する。
/// </summary>
private readonly struct CoreSynthesisResult
{
    public readonly AudioClip Clip;
    public readonly List<PhonemeTimingEntry> Timings;
    public readonly bool HasTimings;

    public CoreSynthesisResult(
        AudioClip clip,
        List<PhonemeTimingEntry> timings)
    {
        Clip = clip;
        Timings = timings;
        HasTimings = timings != null;
    }
}
```

#### SynthesizeWithTimingCoreAsync（新規）

共通コアロジック。推論、正規化、AudioClip 構築、タイミング計算を一括実行する。

```csharp
/// <summary>
/// 共通コア: 音素列から AudioClip + タイミング情報を生成する。
/// SynthesizeAsync と SynthesizeWithTimingAsync の両方から呼ばれる。
/// </summary>
private async Task<CoreSynthesisResult> SynthesizeWithTimingCoreAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default)
{
    if (request.Phonemes == null || request.Phonemes.Length == 0)
        throw new ArgumentException("Phonemes cannot be null or empty.");

    // 1. 音素をIDにエンコード（Prosodyあり/なし）
    int[] phonemeIds;
    int[] expandedProsodyFlat = null;

    if (request.HasProsody)
    {
        var encResult = _phonemeEncoder.EncodeWithProsody(
            request.Phonemes, request.ProsodyFlat);
        phonemeIds = encResult.PhonemeIds;
        expandedProsodyFlat = encResult.ExpandedProsodyFlat;
    }
    else
    {
        phonemeIds = _phonemeEncoder.Encode(request.Phonemes);
    }

    // 2. キャッシュチェック
    long cacheKey = 0;
    if (_audioCache != null)
    {
        cacheKey = AudioSynthesisCache.GenerateKey(
            phonemeIds, expandedProsodyFlat,
            request.LengthScale, request.NoiseScale, request.NoiseW,
            request.SpeakerId, request.LanguageId);

        if (_audioCache.TryGet(
            cacheKey,
            out var cachedSamples,
            out var cachedSampleRate,
            out var cachedTimings))
        {
            PiperLogger.LogInfo(
                "[TTSSynthesisOrchestrator] Cache hit — skipping inference ({0} samples)",
                cachedSamples.Length);
            var tempArray = new NativeArray<float>(cachedSamples, Allocator.Temp);
            try
            {
                var cachedClip = _audioClipBuilder.BuildAudioClip(
                    tempArray, cachedSampleRate,
                    $"TTS_{Guid.NewGuid():N}");
                // cachedTimings は null（タイミングなし）または PhonemeTimingEntry[]
                return new CoreSynthesisResult(
                    cachedClip,
                    cachedTimings != null
                        ? new List<PhonemeTimingEntry>(cachedTimings)
                        : null);
            }
            finally
            {
                tempArray.Dispose();
            }
        }
    }

    // 3. 音声を生成（句分割あり/なし）
    var silenceParsed = _config?.Silence.ParsedPhonemeSilence;
    var useSilenceSplit = _config != null
        && _config.Silence.EnablePhonemeSilence
        && silenceParsed?.Count > 0
        && _voiceConfig?.PhonemeIdMap != null;

    InferenceOutput output = default;
    try
    {
        if (useSilenceSplit)
        {
            output = await _splitOrchestrator.GenerateWithSilenceSplitAsync(
                phonemeIds, expandedProsodyFlat,
                silenceParsed,
                _voiceConfig.PhonemeIdMap,
                _generator.SampleRate,
                request.LengthScale, request.NoiseScale, request.NoiseW,
                request.SpeakerId, request.LanguageId,
                cancellationToken: cancellationToken);
        }
        else
        {
            output = await _generator.GenerateAudioAsync(
                phonemeIds, expandedProsodyFlat,
                request.LengthScale, request.NoiseScale, request.NoiseW,
                request.SpeakerId, request.LanguageId,
                cancellationToken);
        }

        // 4. 正規化
        if (_config == null || _config.Audio.NormalizeAudio)
        {
            AudioNormalizer.NormalizeInPlace(output.Audio, 0.95f);
        }

        // 5. タイミング計算
        List<PhonemeTimingEntry> timings = null;
        if (output.HasDurations && _voiceConfig?.PhonemeIdMap != null)
        {
            timings = CalculateTimings(
                phonemeIds, output.Durations, useSilenceSplit);
        }

        // 6. キャッシュに保存
        if (_audioCache != null && output.Audio.IsCreated)
        {
            _audioCache.Set(
                cacheKey,
                output.Audio.ToArray(),
                _generator.SampleRate,
                timings?.ToArray());
        }

        // 7. AudioClip 構築
        var clip = _audioClipBuilder.BuildAudioClip(
            output.Audio,
            _generator.SampleRate,
            $"TTS_{Guid.NewGuid():N}");

        PiperLogger.LogInfo(
            $"[TTSSynthesisOrchestrator] Synthesized {output.Audio.Length} samples " +
            $"from {request.Phonemes.Length} phonemes" +
            (timings != null ? $", {timings.Count} timing entries" : ""));

        return new CoreSynthesisResult(clip, timings);
    }
    finally
    {
        if (output.Audio.IsCreated || output.HasDurations)
            output.Dispose();
    }
}
```

### 2.4 CalculateTimings ヘルパーメソッド

```csharp
/// <summary>
/// InferenceOutput.Durations から TimingCalculator を呼び出してタイミング情報を算出する。
/// </summary>
/// <param name="phonemeIds">エンコード済み音素ID配列。</param>
/// <param name="durations">モデル出力の durations NativeArray。</param>
/// <param name="usedSilenceSplit">句分割が使用されたかどうか。</param>
/// <returns>タイミングエントリのリスト。句分割使用時は null（P3-2 初期実装）。</returns>
private List<PhonemeTimingEntry> CalculateTimings(
    int[] phonemeIds,
    NativeArray<float> durations,
    bool usedSilenceSplit)
{
    // 句分割使用時: durations は句ごとの結合値であり、
    // 句間無音のオフセットが反映されていないため、正確なタイミング計算が困難。
    // Phase 3 初期実装では句分割時はタイミング情報を提供しない。
    // TODO: 句ごとに TimingCalculator を呼び出し、句間無音オフセットを加算する実装を検討
    if (usedSilenceSplit)
    {
        PiperLogger.LogDebug(
            "[TTSSynthesisOrchestrator] Silence split was used — " +
            "timing calculation skipped (not yet supported for split inference)");
        return null;
    }

    if (!durations.IsCreated || durations.Length == 0)
        return null;

    // NativeArray<float> → float[] に変換（TimingCalculator は managed 配列を受け取る）
    var durationsArray = durations.ToArray();

    var timingEntries = TimingCalculator.Calculate(
        phonemeIds,
        durationsArray,
        _voiceConfig.PhonemeIdMap,
        _puaTokenMapper,
        _generator.SampleRate);

    PiperLogger.LogDebug(
        $"[TTSSynthesisOrchestrator] TimingCalculator: {timingEntries.Count} entries " +
        $"from {phonemeIds.Length} phonemeIds, {durationsArray.Length} durations");

    return timingEntries;
}
```

**句分割時の設計判断**:

P3-2 初期実装では、句分割使用時はタイミング情報を提供しない（`null` を返す）。理由:

1. 句ごとの durations は「句内相対時間」を表し、句間無音のオフセットが含まれていない
2. 正確なタイミングを算出するには、句ごとに TimingCalculator を呼び出し、句間無音の累積オフセットを各 PhonemeTimingEntry の StartSeconds/EndSeconds に加算する必要がある
3. この追加ロジックは P3-2 のスコープを超えるため、将来のフォローアップチケットで対応する
4. 句分割は長文テキスト（沈黙句を含むテキスト）で有効化され、通常のテキストでは句分割なしで推論されるため、多くのユースケースではタイミング情報が利用可能

### 2.5 既存 SynthesizeAsync の変更

内部で `SynthesizeWithTimingCoreAsync` を呼び出し、`AudioClip` のみ返却する。

```csharp
/// <summary>
/// 音素列からAudioClipを生成する（後方互換API）。
/// 内部で SynthesizeWithTimingCoreAsync を呼び出し、AudioClip のみ返却する。
/// </summary>
public async Task<AudioClip> SynthesizeAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default)
{
    var result = await SynthesizeWithTimingCoreAsync(request, cancellationToken);
    return result.Clip;
}
```

**注意**: 既存の `SynthesizeAsync` の外部インターフェース（引数、戻り値）は一切変更しない。内部実装のみ共通コアに委譲する。

### 2.6 SynthesizeWithTimingAsync の追加（internal）

`SynthesisWithTimingResult` を返す新しい internal メソッドを追加する。P4-2 で `PiperTTS.Inference.cs` から呼び出される。

```csharp
/// <summary>
/// 音素列からタイミング情報付き AudioClip を生成する。
/// SynthesisWithTimingResult を返却し、AudioClip とタイミング情報の両方を提供する。
/// </summary>
/// <param name="request">音声合成リクエスト。</param>
/// <param name="cancellationToken">キャンセルトークン。</param>
/// <remarks>
/// durations 非対応モデルまたは句分割使用時は HasTimings == false となる。
/// </remarks>
internal async Task<SynthesisWithTimingResult> SynthesizeWithTimingAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default)
{
    var result = await SynthesizeWithTimingCoreAsync(request, cancellationToken);

    return new SynthesisWithTimingResult(
        result.Clip,
        result.Timings,
        result.Clip != null ? result.Clip.length : 0f);
}
```

### 2.7 キャッシュ呼び出しの変更

P3-3 で `AudioSynthesisCache` が拡張された後、`TryGet` / `Set` のシグネチャが変更される。P3-2 のコードは P3-3 の拡張後のシグネチャを前提とする。

```csharp
// Before (現状):
_audioCache.TryGet(cacheKey, out var cachedSamples, out var cachedSampleRate)
_audioCache.Set(cacheKey, audioData.ToArray(), _generator.SampleRate);

// After (P3-3 拡張後):
_audioCache.TryGet(cacheKey, out var cachedSamples, out var cachedSampleRate, out var cachedTimings)
_audioCache.Set(cacheKey, output.Audio.ToArray(), _generator.SampleRate, timings?.ToArray());
```

**実装順序の依存**: P3-3 が P3-2 より先に完了している必要がある。あるいは P3-2 実装時点で P3-3 が未完了の場合は、キャッシュ呼び出しを暫定的に旧シグネチャで記述し、P3-3 完了後に更新する。依存グラフ上 P3-3 は P1-1 のみに依存するため、P3-2 と並行作業可能。

### 2.8 InferenceOutput の Dispose

`SynthesizeWithTimingCoreAsync` 内で `InferenceOutput` の Dispose を `finally` ブロックで確実に行う。

```csharp
InferenceOutput output = default;
try
{
    // ... 推論・正規化・タイミング計算・AudioClip構築 ...
    return new CoreSynthesisResult(clip, timings);
}
finally
{
    if (output.Audio.IsCreated || output.HasDurations)
        output.Dispose();
}
```

**注意**: `AudioClip.SetData` は NativeArray の内容をコピーするため、`output.Audio` を Dispose しても AudioClip には影響しない。同様に、`durations.ToArray()` で managed 配列にコピー済みのため、`output.Durations` の Dispose も安全。

### 2.9 puaTokenMapper が null の場合

`puaTokenMapper` が null の場合（P3-2 時点の暫定状態、または P4-2 で渡し忘れた場合）、`TimingCalculator.Calculate()` は PUA 逆引きをスキップする（P1-4 で null 安全に設計済み）。音素文字列は phonemeIdMap のキー文字列がそのまま使用され、PUA 文字（`\uE000` 等）がそのまま表示される。機能的には動作するが可読性が低下する。P4-2 で `puaTokenMapper` が正しく渡されることで解消される。

## 5. 懸念事項とレビュー項目

### 5.1 懸念事項

| # | 懸念 | 対策 |
|---|------|------|
| 1 | 既存 `SynthesizeAsync` の振る舞い変更リスク | 内部実装を共通コアに委譲するのみで、外部インターフェースは一切変更しない。既存テストが全 PASS することで後方互換を確認 |
| 2 | `CoreSynthesisResult` の GC アロケーション | `readonly struct` として定義し、ヒープアロケーションを回避。`List<PhonemeTimingEntry>` は参照型だが、タイミング計算は推論パイプラインで1回のみ発生するため問題なし |
| 3 | `puaTokenMapper` が null の場合のタイミング品質 | PUA 逆引きがスキップされ、音素文字列が PUA 文字のまま表示される。機能的には動作する。P4-2 で puaTokenMapper が正しく渡されることで解消 |
| 4 | `durations.ToArray()` のコピーコスト | durations は通常 40-200 要素（160B-800B）であり無視できる。TimingCalculator が managed 配列を受け取る設計のため、コピーは必須 |
| 5 | P3-3 との実装順序 | P3-3（AudioSynthesisCache 拡張）が未完了の場合、キャッシュ呼び出しが旧シグネチャとなりコンパイルエラーが発生する。対策: P3-3 を先に実装するか、暫定的に旧シグネチャで実装して P3-3 完了後に更新 |
| 6 | 句分割時のタイミング非提供 | P3-2 初期実装の制限事項。句分割が有効な場合はタイミング情報が得られない。将来のフォローアップで句ごとの TimingCalculator 呼び出し + オフセット加算を実装する |
| 7 | `SynthesizeAsync` が内部で `CoreSynthesisResult` のタイミング情報を捨てる | タイミング計算のコストは TimingCalculator 呼び出し（数十us）+ List 構築であり、AudioClip 構築や推論時間と比較して無視できる。SynthesizeAsync からの呼び出し時にタイミング計算をスキップする最適化も可能だが、コード複雑化に見合わない |

### 代替案A: SynthesizeAsync と SynthesizeWithTimingAsync を完全独立

### 代替案C: 句分割時もタイミング情報を提供

```csharp
// 句ごとに TimingCalculator を呼び出し、句間無音オフセットを加算
private List<PhonemeTimingEntry> CalculateTimingsWithSilenceSplit(
    List<(int[] PhonemeIds, NativeArray<float> Durations, int SilenceSamples)> phrases)
{
    var allTimings = new List<PhonemeTimingEntry>();
    float cumulativeOffset = 0f;
    foreach (var (phonemeIds, durations, silenceSamples) in phrases)
    {
        var phraseTimings = TimingCalculator.Calculate(
            phonemeIds, durations.ToArray(), ...);
        foreach (var entry in phraseTimings)
        {
            allTimings.Add(new PhonemeTimingEntry(
                entry.Phoneme,
                entry.StartSeconds + cumulativeOffset,
                entry.EndSeconds + cumulativeOffset));
        }
        cumulativeOffset += /* 句の最終時刻 + 無音時間 */;
    }
    return allTimings;
}
```

**メリット**: 句分割使用時もタイミング情報が利用可能。
**デメリット**: SplitInferenceOrchestrator の内部構造（句ごとの phonemeIds と durations の対応）を TTSSynthesisOrchestrator が知る必要がある。現状 SplitInferenceOrchestrator は結合済みの InferenceOutput のみ返し、句ごとの情報を保持しない。SplitInferenceOrchestrator のインターフェース変更が必要。P3-2 のスコープが大幅に拡大する。

**判定**: P3-2 のスコープでは句分割時のタイミング非提供とし、将来のフォローアップチケットで対応する。フォローアップでは SplitInferenceOrchestrator のインターフェース拡張（句ごとの情報を保持する戻り値型の導入）を検討する。

## 7. 後続タスクへの連絡事項

### P3-3 (AudioSynthesisCache タイミング格納拡張) への連絡

- P3-2 は P3-3 の拡張後シグネチャ（`TryGet` に `out PhonemeTimingEntry[] timings`、`Set` に `PhonemeTimingEntry[] timings` パラメータ）を前提としている
- P3-3 が先に完了していない場合、P3-2 のキャッシュ呼び出し箇所がコンパイルエラーになる
- 推奨実装順序: P3-3 → P3-2、または並行作業の場合は暫定シグネチャで対応

### P4-1 (IPiperTTS インターフェース拡張) への連絡

- `TTSSynthesisOrchestrator.SynthesizeWithTimingAsync()` が `internal` メソッドとして実装済み
- `IPiperTTS` に `SynthesizeWithTimingAsync()` メソッドを追加し、PiperTTS 実装で TTSSynthesisOrchestrator.SynthesizeWithTimingAsync() を委譲すること
- 戻り値型: `Task<SynthesisWithTimingResult>`

### P4-2 (PiperTTS.Inference.cs 実装) への連絡

- `TTSSynthesisOrchestrator` のコンストラクタに `puaTokenMapper` パラメータが追加されている
- `PiperTTS.Inference.cs` で TTSSynthesisOrchestrator を構築する際に、PiperTTS が保持する `PuaTokenMapper` インスタンスを渡すこと
- `SynthesizeWithTimingAsync()` は `internal` メソッドのため、同一アセンブリ内からアクセス可能

### T3-3 (durations 非対応モデルテスト) への連絡

- StubInferenceAudioGenerator の `SupportsDurations` / `DurationsToReturn` を制御してテスト
- durations なし設定: `DurationsToReturn = null` → `SynthesizeWithTimingAsync` で `HasTimings == false`
- durations あり設定: `DurationsToReturn = new float[] { ... }` → `HasTimings == true`
- 句分割テスト: `IPiperConfigReadOnly.Silence.EnablePhonemeSilence = true` + phoneme_silence 設定 → `HasTimings == false`（P3-2 初期実装の制限）

### 将来フォローアップ: 句分割時タイミング対応

- P3-2 では句分割使用時にタイミング情報を提供しない（`CalculateTimings` が `null` を返す）
- フォローアップで対応する場合の設計方針:
  1. `SplitInferenceOrchestrator` のインターフェースを拡張し、句ごとの phonemeIds + durations + silenceSamples のリストを返す新しいメソッドを追加（または戻り値型を拡張）
  2. `TTSSynthesisOrchestrator.CalculateTimings` で句ごとに TimingCalculator を呼び出し、句間無音の累積オフセットを加算
  3. `PhonemeTimingEntry` のコンストラクタは `internal` のため、オフセット加算後の新しいエントリを構築可能
