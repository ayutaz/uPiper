# T4-5: リグレッションテスト

## 1. タスク目的とゴール

Phase 4（SynthesizeWithTimingAsync 追加）後に既存の SynthesizeAsync / GenerateAudioAsync の動作に変更がないことを検証する。フォネムタイミング機能の追加により、TTSSynthesisOrchestrator の内部構造が2段構成（SynthesizeWithTimingCoreAsync → SynthesizeAsync/SynthesizeWithTimingAsync）に変更されるため、既存 API のリグレッションがないことを保証する。

**ゴール**: 既存の SynthesizeAsync() が引き続き AudioClip を返すこと、GenerateAudioAsync() のパラメータ伝播が正常であること、ガード句（disposed, not-initialized）の動作が不変であること、ClearCache() がタイミング情報を含めてクリアすることを保証する。

### コードパターン例

```csharp
// PiperTTSSynthesizeWithTimingTests.cs に追加
[Test]
public void SynthesizeWithTimingAsync_NotInitialized_ThrowsInvalidOperationException()
{
    using var tts = new PiperTTS(_config);
    var request = SynthesisRequest.FromPhonemes(new[] { "a", "b" });

    var ex = Assert.ThrowsAsync<InvalidOperationException>(
        async () => await tts.SynthesizeWithTimingAsync(request));
    StringAssert.Contains("not initialized", ex.Message,
        "Should mention not initialized");
}

[Test]
public void SynthesizeWithTimingAsync_Disposed_ThrowsObjectDisposedException()
{
    var tts = new PiperTTS(_config);
    tts.Dispose();

    var request = SynthesisRequest.FromPhonemes(new[] { "a", "b" });

    Assert.ThrowsAsync<ObjectDisposedException>(
        async () => await tts.SynthesizeWithTimingAsync(request));
}

// TTSSynthesisOrchestratorTests.cs に追加
[Test]
public async Task SynthesizeAsync_AfterTimingFeature_StillReturnsAudioClipOnly()
{
    // Arrange: durations 対応モデルでも SynthesizeAsync は AudioClip のみ返す
    _stubGenerator.DurationsToReturn = new float[9];
    for (var i = 0; i < 9; i++)
        _stubGenerator.DurationsToReturn[i] = 10.0f;

    var config = CreateValidatedConfig(enableSilence: false);
    var orchestrator = new TTSSynthesisOrchestrator(
        _stubGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
        config, _voiceConfig);
    var request = new SynthesisRequest(
        new[] { "k", "o", "N" },
        null,
        1.0f, 0.667f, 0.8f,
        0, 0);

    // Act
    var clip = await orchestrator.SynthesizeAsync(request);

    // Assert
    Assert.IsNotNull(clip, "SynthesizeAsync は AudioClip を返すこと");
    Assert.IsInstanceOf<AudioClip>(clip,
        "戻り値が AudioClip 型であること");
}
```

## 6. 一から作り直すとしたら

リグレッションテストの核心は「既存テストが変更なしで PASS し続けること」。新規テストメソッドの追加は最小限に抑え、以下の 3 カテゴリに集中する:

1. **ガード句テスト（2メソッド）**: SynthesizeWithTimingAsync が SynthesizeAsync と同等のガード句を持つことの保証。PiperTTSSynthesizeTests.cs の既存パターンを完全に踏襲する
2. **動作不変性テスト（2メソッド）**: SynthesizeAsync の戻り値型と パラメータ伝播がタイミング機能追加後も不変であることの明示的な保証
3. **キャッシュ互換性テスト（1メソッド）**: SynthesizeWithTimingAsync と SynthesizeAsync がキャッシュを共有できることの保証。これは新機能の追加による予期せぬ相互作用がないことの検証

既存テスト（PiperTTSSynthesizeTests 6メソッド + TTSSynthesisOrchestratorTests 17メソッド = 23メソッド）の PASS 確認は CI パイプラインに委ね、本チケットでは「既存テストの実行結果を確認する」のみとする。既存テストの変更は各実装チケット（P3-2, P4-2）の責務。
