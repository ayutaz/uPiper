# P4-2: PiperTTS.Inference.cs 実装

## 1. タスク目的とゴール

`PiperTTS.Inference.cs` に `SynthesizeWithTimingAsync` の正式実装を追加し、P4-1 で追加した暫定スタブ（`NotImplementedException`）を置き換える。これにより、エンドユーザーが `IPiperTTS.SynthesizeWithTimingAsync` を呼び出してタイミング情報付き音声合成結果を取得できるようになる。

**ゴール**: `PiperTTS.SynthesizeWithTimingAsync` が `TTSSynthesisOrchestrator.SynthesizeWithTimingAsync` に正しく委譲し、Dispose/初期化チェックを行い、既存の `SynthesizeAsync` に影響を与えないこと。

### なぜ必要か

- P4-1 で `IPiperTTS` にメソッド宣言が追加されたが、実装は暫定スタブ（`NotImplementedException`）のまま
- P3-2 で `TTSSynthesisOrchestrator.SynthesizeWithTimingAsync`（internal）が実装済みだが、公開 API からの呼び出し経路がない
- このチケットで公開 API → internal オーケストレーター の接続を完成させ、フォネムタイミング機能のエンドツーエンドパイプラインが完結する

### 2.2 SynthesizeWithTimingAsync 正式実装

P4-1 で追加した暫定スタブを以下の正式実装に置き換える。既存の `SynthesizeAsync` のパターンを踏襲する。

```csharp
/// <summary>
/// SynthesisRequestを直接指定してタイミング情報付き音声を生成する（低レベルAPI）。
/// 音素列は事前に <see cref="PhonemizeAsync"/> または外部G2Pで取得・構築済みであること。
/// </summary>
/// <param name="request">音声合成リクエスト（音素・Prosody・合成パラメータを集約）。</param>
/// <param name="cancellationToken">キャンセルトークン。</param>
/// <returns>
/// タイミング情報付き音声合成結果。
/// <see cref="SynthesisWithTimingResult.HasTimings"/> が false の場合、
/// モデルが durations テンソルを出力しない（非対応モデル）か、
/// 句分割推論が使用されたことを意味する。その場合でも AudioClip は有効。
/// </returns>
/// <exception cref="ObjectDisposedException">インスタンスがDispose済みの場合。</exception>
/// <exception cref="InvalidOperationException">Inferenceが初期化されていない場合。</exception>
public async Task<SynthesisWithTimingResult> SynthesizeWithTimingAsync(
    SynthesisRequest request,
    CancellationToken cancellationToken = default)
{
    if (_isDisposed)
        throw new ObjectDisposedException(nameof(PiperTTS));

    if (!_isInitialized || _orchestrator == null)
        throw new InvalidOperationException(
            "Inference is not initialized. Call InitializeWithInferenceAsync first.");

    return await _orchestrator.SynthesizeWithTimingAsync(request, cancellationToken);
}
```

### 2.3 既存 SynthesizeAsync との比較

新旧メソッドの対照表:

| 項目 | SynthesizeAsync（既存） | SynthesizeWithTimingAsync（新規） |
|------|------------------------|----------------------------------|
| 戻り値型 | `Task<AudioClip>` | `Task<SynthesisWithTimingResult>`（修飾なし、`using uPiper.Core.AudioGeneration;` 利用） |
| Dispose チェック | `_isDisposed` → `ObjectDisposedException` | 同一パターン |
| 初期化チェック | `!_isInitialized \|\| _orchestrator == null` | 同一パターン |
| 委譲先 | `_orchestrator.SynthesizeAsync(request, ct)` | `_orchestrator.SynthesizeWithTimingAsync(request, ct)` |
| エラーラップ | なし（例外はそのまま伝搬） | なし（同一パターン） |

**設計判断**: 既存 `SynthesizeAsync` が例外をラップせず直接伝搬するパターンを踏襲する。`GenerateAudioWithInferenceAsync`（Obsolete）は `PiperException` でラップしているが、新しい低レベル API（`SynthesizeAsync` / `SynthesizeWithTimingAsync`）は例外を素通しさせて呼び出し元に正確なスタックトレースを提供する。

### 2.4 TTSSynthesisOrchestrator コンストラクタへの _tokenMapper 追加

P3-2 で `TTSSynthesisOrchestrator` のコンストラクタに `PuaTokenMapper puaTokenMapper = null` パラメータが追加される。`PiperTTS.Inference.cs` の `InitializeWithInferenceAsync` 内でオーケストレーターを構築する箇所を変更し、`_tokenMapper` を渡す。

**変更前**（現状）:
```csharp
_orchestrator = new TTSSynthesisOrchestrator(
    mitigatingGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
    _validatedConfig, voiceConfig, _audioSynthesisCache);
```

**変更後**:
```csharp
_orchestrator = new TTSSynthesisOrchestrator(
    mitigatingGenerator, _splitOrchestrator, _phonemeEncoder, _audioClipBuilder,
    _validatedConfig, voiceConfig, _audioSynthesisCache, _tokenMapper);
```

これにより `TimingCalculator` が PUA 逆引きを行い、`PhonemeTimingEntry.Phoneme` に可読な音素文字列（`ch`, `ts` 等）が格納されるようになる。`_tokenMapper` は `PiperTTS` クラスのフィールドとして既に存在する（`PiperTTS.cs` L54: `private readonly Phonemizers.Multilingual.PuaTokenMapper _tokenMapper = new();`）。

## 6. 一から作り直すとしたら

現設計は適切である。既存 `SynthesizeAsync` と完全に対称的なパターンで実装されており、学習コストが最小限。再設計するとしても以下の判断は維持する:

- **委譲パターン**: `PiperTTS` はファサードとして Dispose/初期化チェックのみ行い、実質的な処理は `TTSSynthesisOrchestrator` に委譲する。このレイヤリングにより、単体テストが容易になる
- **例外を素通し**: 低レベル API では `PiperException` ラップを行わず、オーケストレーター以下の例外をそのまま伝搬する。デバッグ時のスタックトレースが正確になる
- **`_tokenMapper` の受け渡し**: `PiperTTS` が所有する `_tokenMapper` をオーケストレーターに渡すことで、PUA 逆引きが可能になる。別途 DI コンテナを導入するほどの複雑さはない

変更を検討する点:
- `SynthesizeAsync` の内部実装も `SynthesizeWithTimingAsync` → `.AudioClip` で統一する案があるが、P3-2 で既にオーケストレーター内部で共通コア化されているため、`PiperTTS` レベルでの統合は不要（むしろ既存メソッドの変更リスクが増える）
