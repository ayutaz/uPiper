# T3-1: SplitInferenceOrchestrator 句分割タイミングテスト

## 1. タスク目的とゴール

SplitInferenceOrchestrator が複数句の InferenceOutput（Audio + Durations）を結合した際に、結合後の Audio.Length と Durations.Length が正しいこと、および句間無音が Audio にのみ挿入され Durations には含まれないことを検証する。P3-1 で SplitInferenceOrchestrator の戻り値が `NativeArray<float>` から `InferenceOutput`（Audio + Durations）に変更され、句ごとの durations を結合するロジックが追加される。本チケットではそのロジックの正確性を、StubInferenceAudioGenerator の `DurationsToReturn` 経由で固定 durations を注入して検証する。

**ゴール**: 2句以上の句分割時に、結合後の Audio 長が「各句のAudio + 句間無音サンプル」の合計と一致し、結合後の Durations 長が「各句のDurations」の合計と一致すること（句間無音に対応する duration エントリは含まない）を保証する。

### 前提知識

P3-1 で SplitInferenceOrchestrator に追加される変更:
- `GenerateWithSilenceSplitAsync` の戻り値が `NativeArray<float>` から `InferenceOutput` に変更
- 句ごとに `_generator.GenerateAudioAsync()` が `InferenceOutput` を返す
- Audio は既存ロジックと同様に結合（句間無音サンプル挿入）
- Durations は各句の durations を連結（句間無音に対応する duration エントリは不要 -- 無音はサンプル数ベースで Audio に直接挿入されるため）

P2-4 で StubInferenceAudioGenerator に追加される変更:
- `DurationsToReturn` プロパティ（`float[]` 型）で固定 durations データを注入可能
- `GenerateAudioAsync` の戻り値が `Task<InferenceOutput>` に変更

### 実装するテストメソッド

1. **GenerateWithSilenceSplitAsync_TwoPhrases_CombinedAudioLengthCorrect**
   - Arrange: phonemeIds = [1, 3, 0, 8, 7, 2]（`_` で2句に分割）
   - StubGenerator: AudioDataToReturn = float[50]、DurationsToReturn = float[3]（各句3音素分）

   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: `output.Audio.Length == 50 + 11025 + 50`（句1 + 無音 + 句2）
   - try/finally で InferenceOutput.Dispose()

2. **GenerateWithSilenceSplitAsync_TwoPhrases_CombinedDurationsLengthCorrect**
   - Arrange: 同上（2句分割）
   - StubGenerator: DurationsToReturn = float[3]（[1.0f, 2.0f, 3.0f]）
   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: `output.Durations.Length == 6`（句1の3要素 + 句2の3要素）
   - Assert: `output.HasDurations == true`
   - try/finally で Dispose

3. **GenerateWithSilenceSplitAsync_TwoPhrases_DurationsValuesPreserved**
   - Arrange: 同上（2句分割）
   - StubGenerator: DurationsToReturn = float[3]（[10.0f, 20.0f, 30.0f]）
   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: 結合後の Durations が [10, 20, 30, 10, 20, 30] の順で値が保持されていること
   - StubGenerator は毎回同じ DurationsToReturn を返すため、句1/句2とも同じ値になる
   - try/finally で Dispose

4. **GenerateWithSilenceSplitAsync_ThreePhrases_SilenceOffsetAccumulated**
   - Arrange: phonemeIds = [1, 3, 0, 8, 0, 7, 2]（`_` で3句に分割）
   - phonemeSilence = { "_": 0.2f } → 無音 = 0.2 * 22050 = 4410 サンプル
   - StubGenerator: AudioDataToReturn = float[30]、DurationsToReturn = float[2]
   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: `output.Audio.Length == 30 + 4410 + 30 + 4410 + 30`（3句 + 2区間の無音）
   - Assert: `output.Durations.Length == 6`（3句 * 2要素）
   - 句間無音オフセットが Audio 長に正しく累積されていること
   - try/finally で Dispose

5. **GenerateWithSilenceSplitAsync_SinglePhrase_DurationsUnchanged**
   - Arrange: phonemeIds = [1, 3, 4, 5, 2]（分割なし）
   - StubGenerator: DurationsToReturn = float[5]（[1.0f, 2.0f, 3.0f, 4.0f, 5.0f]）
   - phonemeSilence = { "#": 0.5f }（phonemeIds に "#" の ID が存在しないため分割されない）
   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: `output.Durations.Length == 5`（分割なしなので変更なし）
   - Assert: 各要素が [1.0f, 2.0f, 3.0f, 4.0f, 5.0f] と一致
   - try/finally で Dispose

6. **GenerateWithSilenceSplitAsync_NoDurations_HasDurationsFalse**
   - Arrange: phonemeIds = [1, 3, 0, 8, 7, 2]（2句分割）
   - StubGenerator: DurationsToReturn = null（durations 非対応モデル）
   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: `output.HasDurations == false`
   - Audio の結合は正常に行われること（Audio.Length は句のオーディオ + 無音の合計）
   - try/finally で Dispose

7. **GenerateWithSilenceSplitAsync_EmptyInput_NoDurations**
   - Arrange: phonemeIds = Array.Empty<int>()
   - StubGenerator: DurationsToReturn = float[5]（設定されているが使われない）
   - Act: `_orchestrator.GenerateWithSilenceSplitAsync(...)`
   - Assert: `output.Audio.Length == 0`
   - Assert: `output.HasDurations == false`（空入力では推論が呼ばれないため durations も空）

### SetUp の変更

既存の `[SetUp]` メソッドでは `_stubGenerator.AudioDataToReturn` のみ設定している。durations テスト用に `DurationsToReturn` を併せて設定するか、各テストメソッド内で個別に設定する。既存テストとの互換性を保つため、`[SetUp]` では `DurationsToReturn = null`（デフォルト）のままとし、各テストメソッド内で必要に応じて設定するパターンを推奨する。

### コードパターン例

```csharp
[Test]
public async Task GenerateWithSilenceSplitAsync_TwoPhrases_CombinedDurationsLengthCorrect()
{
    // Arrange
    _stubGenerator.DurationsToReturn = new float[] { 1.0f, 2.0f, 3.0f };
    var phonemeIds = new[] { 1, 3, 0, 8, 7, 2 };
    var phonemeSilence = new Dictionary<string, float> { ["_"] = 0.5f };
    var phonemeIdMap = CreateMinimalPhonemeIdMap();

    // Act
    var output = await _orchestrator.GenerateWithSilenceSplitAsync(
        phonemeIds,
        prosodyFlat: null,
        phonemeSilence, phonemeIdMap,
        sampleRate: 22050);

    try
    {
        // Assert
        Assert.IsTrue(output.HasDurations, "durations が存在すること");
        Assert.AreEqual(6, output.Durations.Length,
            "2句 * 3要素 = 6 の結合 durations が返ること");
    }
    finally
    {
        output.Dispose();
    }
}
```

## 6. 一から作り直すとしたら

テスト設計は「durations 結合の入出力検証」に絞るのが最適。SplitInferenceOrchestrator の Audio 結合ロジック自体は既存テスト（6テストメソッド）で十分にカバーされているため、本チケットでは durations の結合に特化する。StubInferenceAudioGenerator が固定 DurationsToReturn を返す制約は、結合ロジック自体の検証には十分。句ごとに異なる durations が必要なケース（例: 異なる長さの句で durations の要素数が異なる）は、スタブの拡張（呼び出し回数に応じて異なるデータを返す `Queue<float[]>` パターン）と合わせて別チケットで対応するのが妥当。

句間無音が Durations に含まれない設計は、「durations は推論モデルの出力であり、後処理で挿入される無音は durations の対象外」という原則に基づく。テストではこの原則を `Audio.Length != (sum of durations * frameLength * sampleRate)` の非対称性として明示的に検証する。

## 7. 後続タスクへの連絡事項

- P3-1 実装時: `GenerateWithSilenceSplitAsync` の戻り値を `InferenceOutput` に変更する際、Audio 結合と同様に Durations も `NativeArray<float>.Copy` で結合すること。句間無音に対応する duration エントリは挿入しない（Audio のみ無音サンプルを追加）
- P3-1 実装時: いずれかの句の `phraseOutput.HasDurations` が true の場合に `hasDurations` フラグを true にし、結合 durations を生成すること。`SupportsDurations` はモデル単位のプロパティであり全句が同一状態を持つため、一部のみ durations を持つ不整合は想定外（P3-1 セクション 2.7 参照）
- P2-4 実装時: StubInferenceAudioGenerator の DurationsToReturn が各呼び出しで同一データを返す点を文書化すること。本チケットのテストはこの前提に基づく
- T3-3（durations 非対応モデルテスト）: 本チケットのテスト #6 で SplitInferenceOrchestrator レベルの durations 非対応をカバーするが、TTSSynthesisOrchestrator レベルのテストは T3-3 で実施
- T4-1（統合テスト）: SplitInferenceOrchestrator → TimingCalculator の連携テストは T4-1 で実施。本チケットでは TimingCalculator を使用しない
